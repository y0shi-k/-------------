# Reporting

## ユーザー向け報告の順番
1. 今回の変更目的
2. これまで不足していた点
3. 今回追加した安全装置
4. 実施した確認
5. まだ人判断が必要な残リスク

## 書き方
- 先に「何の事故を減らす変更か」を書く
- 専門用語は意味を添える
- 実装詳細やファイル名は必要最小限にする
- 失敗時は「どこで止まったか」と「何を守るために止めたか」を平易に書く
- required gate が未達なら、完了表現を使わず open gate 名をそのまま書く

## artifact
- 下書きは `project-os/artifacts/TKT-xxxx/report.md` に残す
- `project-os/artifacts/TKT-xxxx/` 配下に残す `verify.json` / `manual-smokes.md` / `review.md` / `report.md` は、ユーザーが読む前提で必ず日本語で記述する。
- コマンド名、ファイルパス、status値、API名などの機械的な識別子は英語のままでよいが、説明文・判断理由・残リスク・次の作業は日本語にする。
- artifact の本文を英語で作成した場合は、完了前に日本語へ直す。

## 軽量UI調整の報告
- 同一 ticket に集約した軽量UI調整は、小修正ごとに個別報告を分けず、一連の変更としてまとめて報告する。
- ブラウザテストをユーザーが実施する場合は、「AIが実施した静的確認」と「ユーザーに残した手動確認項目」を分けて書く。
- required gate を軽量集約で最後にまとめて閉じる場合、途中経過では「完了」ではなく「静的確認済み」「ユーザー確認待ち」などの状態表現を使う。
