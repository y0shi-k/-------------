# 料理レシピ・食材管理アプリ 要件定義書（AI開発指示書）

本ドキュメントは「料理レシピ・食材管理アプリ」の完全な要件定義書です。 開発を行うAIエージェントは、本ドキュメントの「データスキーマ」および「機能要件」に厳密に従い、勝手な解釈による仕様変更やカラムの追加・削除を行わずにコードを生成してください。

## 1. システムアーキテクチャ・技術要件

*   **バックエンド:** Google Apps Script (GAS)
    
*   **データベース:** Google Spreadsheet (1つのファイル内に複数シート構成)
    
*   **フロントエンド:** 1枚のHTMLファイル (Tailwind CSS, CDN経由のJavaScript利用)
    
*   **UI構造:** 画面下部に固定のボトムナビゲーションバーを設置し、「食材管理（モードA）」「献立・レシピ（モードB）」「料理・記録（モードC）」をワンタップでシームレスに切り替えられるSPA（Single Page Application）構成とする。
    
*   **UX/フェイルセーフ（重要）:** * GASとの通信（API呼び出し、データ取得・保存等）を行う際は、必ず実行ボタンを無効化（disabled）し、二重クリックを防止すること。
    
    *   **詳細なステータスバー表示:** 単なる「処理中...」やスピナーだけでなく、画面上部または中央にステータスバー（プログレスインジケーター）を設置し、裏側で何を行っているのかを細かくリアルタイムに表示すること。（例：「画像をGemini APIに送信中...」→「AIが食材を解析中...」→「スプレッドシートに保存中...」→「完了！」）。これによりユーザーの待ち時間の体感ストレスを軽減する。
        
    *   また、通信エラーや処理失敗時には、画面上にエラー理由をわかりやすく表示し、フリーズしたように見せない設計とすること。
        
*   **AI連携:** Gemini API (マルチモーダル画像解析、テキスト構造化)
    
*   **永続化:** アプリ設定やファイルIDは `PropertiesService` に保存し、GAS実行ごとにシートが増殖しないように初期化処理を実装すること。
    

## 2. データベース設計（スプレッドシートスキーマ）

初期化時に以下の5つのシートを作成し、1行目に厳密に以下のヘッダー（カラム名）を設定すること。

### ① 食材在庫

*   A: `ID` (String: UUID)
    
*   B: `分類` (String: 食材 / 調味料)
    
*   C: `品名` (String)
    
*   D: `数量` (Number)
    
*   E: `単位` (String: 個, g, ml, 本 など)
    
*   F: `購入日` (String: YYYY-MM-DD)
    
*   G: `開封日` (String: YYYY-MM-DD または空)
    
*   H: `表示期限` (String: YYYY-MM-DD) ※パッケージ印字の賞味/消費期限
    
*   I: `実質期限` (String: YYYY-MM-DD または空) ※開封日を元にAIが推測、またはユーザーが設定した実際の消費目安
    
*   J: `保存場所` (String: 冷蔵庫 / 冷凍庫 / パントリー / ベランダ / その他)
    
*   K: `状態メモ` (String)
    

### ② レシピ集

*   A: `レシピID` (String: UUID)
    
*   B: `レシピ名` (String)
    
*   C: `材料JSON` (String: `[{"name":"人参", "amount": 1, "unit": "本"}, ...]`)
    
*   D: `手順JSON` (String: `["手順1", "手順2", ...]`)
    
*   E: `出典` (String: URLまたはテキスト)
    
*   F: `調理回数` (Number: 初期値0)
    
*   G: `調理日履歴` (String: JSON配列 `["YYYY-MM-DD", ...]`) ※いつ調理したかの履歴リスト
    

### ③ 献立スケジュール

*   A: `予定日` (String: YYYY-MM-DD)
    
*   B: `食事区分` (String: 朝 / 昼 / 晩 / その他)
    
*   C: `レシピID` (String: UUID)
    
*   D: `レシピ名` (String)
    
*   E: `ステータス` (String: 未完了 / 完了)
    

### ④ 料理履歴（実行ログ）

*   A: `履歴ID` (String: UUID)
    
*   B: `調理日時` (String: YYYY-MM-DD HH:mm:ss)
    
*   C: `レシピID` (String: UUID)
    
*   D: `レシピ名` (String)
    
*   E: `感想` (String)
    
*   F: `写真URL` (String: DriveAppで保存したファイルURL)
    
*   G: `評価` (Number: 1〜5)
    

### ⑤ 買い物リスト

*   A: `リストID` (String: UUID)
    
*   B: `品名` (String)
    
*   C: `必要数量` (Number)
    
*   D: `単位` (String)
    
*   E: `ステータス` (String: 未購入 / 購入済)
    
*   F: `紐づくレシピ名` (String)
    

## 3. 機能要件詳細

### モードA: 食材インベントリ（管理・スキャン・買い物モード）

*   **機能概要:** 現在庫の確認、フル編集、AI一括登録、および買い物リストの確認を行うダッシュボード。
    
*   **UI要件:**
    
    *   「保存場所」ごとのタブ表示に並べて、**「買い物リスト」タブ**を設置する。
        
    *   アラート表示: 在庫タブでは `実質期限`（未設定の場合は `表示期限`）を基準とし、期限が3日以内のものは黄背景、期限切れは赤背景でハイライト。
        
    *   一覧の各行に「＋」「－」ボタンを配置し、ワンクリックで数量を増減（消費）可能。0になった場合は行を削除するか尋ねる。
        
    *   「編集」ボタンから全項目（品名や開封日など）を手動変更できるモーダル。
        
    *   「手動追加」ボタンから新規食材を登録できるフォーム。
        
*   **買い物リスト機能（タブ内）:**
    
    *   シート⑤の未購入アイテムをチェックボックス付きのリストで表示。
        
    *   スーパーでの買い回り中にチェックを入れ、「購入済にする」ボタンを押すと、チェックしたアイテムがそのまま「食材在庫」シートにワンタップで移行（新規登録・または既存在庫に加算）される機能。
        
*   **AIスキャナー＆推測処理:**
    
    *   **画像スキャン:** カメラ撮影または画像アップロード画像をGemini APIに送信しデータを抽出。`[{"分類":"", "品名":"", "数量":0, "単位":"", "推測表示期限":"YYYY-MM-DD", "推測保存場所":""}]` のフォーマットで「食材在庫」シートへ一括追記する。
        
    *   **開封後の実質期限推測:** ユーザーが一覧や編集画面から「開封日」を入力（または今日を開封日に設定）した際、Gemini APIがその食材名（例: めんつゆ、牛乳）と開封日から「開封後の実質的な消費期限（実質期限）」を自動計算して設定する。
        

### モードB: スマート献立プランナー

*   **機能概要:** 献立管理UI、AIレシピ考案、テキストからの抽出、および買い物リスト連携。
    
*   **UI/UXデザイン要件（スケジュール表示）:**
    
    *   **週間リストビュー形式:** 小さなマス目の月間カレンダーではなく、スマホでも見やすい縦スクロールのリスト形式を採用する。
        
    *   **朝・昼・晩の独立スロット:** 各日付カード内に「朝」「昼」「晩」の枠を一覧表示し、1日の予定がパッと見てわかるようにする。
        
    *   **簡単割り当て:** スロットが空の場合は「＋追加」ボタンを表示し、タップでレシピをセットできるようにする。
        
*   **手動/自動割り当て:** 「レシピ集」から選んで日付・スロットにセット。またはAIに「この日の献立」を提案させる。
    
*   **AIレシピ考案・抽出要件:**
    
    *   **在庫優先提案:** 「食材在庫」の実質期限（または表示期限）が迫っているデータをプロンプトに渡し、優先消費レシピを生成。
        
    *   **指定食材提案:** ユーザーが在庫から選んだ複数アイテム（例: 豚肉と白菜）を必須要件としてレシピ生成。
        
    *   **外部テキスト抽出:** Webのコピーやメモをテキストエリアに貼り付け、Gemini APIで解析。「レシピ名」「材料JSON」「手順JSON」に厳密に構造化して抽出。
        
*   **連携処理とジャンプ導線（買い物リスト）:**
    
    *   献立決定時やAIレシピ確定時、レシピの`材料JSON`と現在の`食材在庫`を自動で比較する。
        
    *   在庫がない、または足りないアイテムを抽出し、「買い物リスト」シートに自動追記する。
        
    *   追加が完了した直後、画面上に「不足する材料が買い物リストに追加されました
        
        リストを確認する
        
        」というジャンプボタン（導線）を表示し、タップすると直接モードAの「買い物リスト」タブへ画面遷移できるようにする。
        

### モードC: クッキングビューア＆食卓メモリー（実行・記録統合モード）

*   **機能概要:** 料理中の大画面ビュー、消費量の微調整、在庫減算、写真・感想記録をシームレスに行う。
    
*   **UI/UXデザイン要件（最重要）:**
    
    *   **デカ文字・大画面:** 遠くからでも読める大きなフォントサイズ（Tailwindの `text-xl` などを多用）。
        
    *   **視覚的分離:** 「食材（肉・野菜等）」は緑系の背景バッジ、「調味料」は黄色系の背景バッジで表示を分ける。
        
    *   **大さじ・小さじの絶対区別:** 計量ミス防止のため、「大さじ」は【赤色バッジ＋太字】、「小さじ」は【青色バッジ＋太字】で表示。
        
    *   **インライン分量表示:** 手順テキスト内にも分量を強調表示する（例: 「醤油 大さじ1 を加える」）。
        
*   **実行・記録フロー:**
    
    1.  レシピ通りに料理を進め、最後に「完了ボタン」を押す。
        
    2.  **消費量調整モーダル:** レシピの規定量がセットされた状態で表示。「実際に使った量」をユーザーが手動で微調整できる（多く使った、余らせた等）。
        
    3.  確定ボタンで、調整された数量を「食材在庫」から減算する。
        
    4.  **記録モーダル:** そのまま「完成写真のアップロード」「評価（★1〜5）」「感想」の入力画面へ遷移。
        
    5.  送信すると、「料理履歴」に追記、「献立スケジュール」のステータスを完了に変更、「レシピ集」の調理回数と**調理日履歴**を更新する。
        

## 4. AI開発エージェントへの指示（開発フェーズ分割）

開発は複雑さを避けるため、必ず以下のフェーズごとに分けてプロンプトを受け取り、実装を進めること。

*   Phase1
    
    データベース構築 ＋ モードA（インベントリ・買い物リスト）
    
    *   `PropertiesService`を用いたスプレッドシートの初期化・ヘッダー作成処理。
        
    *   在庫の読み込み表示、手動編集機能、画像（Geminiマルチモーダル）からのAI一括登録機能、開封日入力時の「実質期限」AI自動推測機能。
        
    *   買い物リストタブのUI構築と、購入済みアイテムの在庫へのワンタップ移行機能。
        
    *   画面下部ボトムナビゲーションバーの基礎UI構築。
        
*   Phase2
    
    モードB（レシピ集と献立プランナー）
    
    *   AIによるレシピ考案、テキストからのレシピ構造化機能。
        
    *   朝・昼・晩スロットを持つ週間リストカレンダーの構築。
        
    *   献立割り当て時の「買い物リスト」自動抽出ロジックと、買い物リストタブへのジャンプ導線の実装。
        
*   Phase3
    
    モードC（クッキングビューアと記録）
    
    *   色分け・バッジ化された大画面UIの構築。
        
    *   完了後の消費量調整UIと在庫減算処理。
        
    *   画像アップロード（Drive保存）を伴う料理履歴の保存機能。
