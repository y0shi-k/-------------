# Gemini Canvas (プレビュー環境) 開発の注意点・制約リスト

Geminiの右画面（Canvas/Immersive環境）でWebアプリを開発・動作させる上で、留意すべき特有の仕様や制限を網羅的にまとめました。

## 1. APIキーと通信の挙動（先ほどの件）

*   **APIキーの自動解決:** Canvas内でGemini API等の特定のエンドポイントを叩く際、システム側が環境の裏側で自動的にリクエストをプロキシし、キーを補完する場合があります。そのため、コード上で `if (!API_KEY) { return エラー; }` のような厳密な空チェックを入れると、かえって処理がブロックされることがあります。
    
*   **CORS（Cross-Origin Resource Sharing）の制限:** Canvasはセキュリティ上、特殊なドメインのiframe内で実行されます。そのため、CORS設定が厳格な外部APIへ `fetch()` を行うとエラーになる確率が高いです。（※現在のアプリで採用している、GASへform送信＋スクリプトタグでコールバックを受け取るJSONP的な手法は、この制限をうまく回避できる非常に優れた設計です）
    

## 2. 🚨 【超重要】標準UIダイアログの利用禁止

*   **`alert()`, `confirm()`, `prompt()` は使用NG:** Canvasのiframe内でこれらのネイティブダイアログを呼び出すと、ブラウザの仕様によりプレビュー画面やUI全体がフリーズ（ロック）してしまう不具合を引き起こす危険性があります。
    
*   **現状のコードへのフィードバック:** 現在のコード内に `alert('データの読み込みに失敗しました');` や `confirm('リストから削除しますか？')` という記述がいくつか存在しています。Canvas環境を安定させるため、これらは既に実装されている `showToast()` を活用するか、HTMLベースのカスタムモーダル（「はい/いいえ」ボタン付きのポップアップ）に置き換えることを強く推奨します。
    

## 3. アセット（画像や音声）の読み込み制限

*   **ローカルファイルは参照不可:** `<img src="./images/apple.png">` のような相対パスでのローカルアセット読み込みは機能しません。
    
*   **代替手段:**
    
    1.  外部のパブリックURL（CORS対応の画像CDNなど）を指定する。
        
    2.  プレースホルダー画像サービス（ `https://placehold.co/` など）を利用する。
        
    3.  アイコンなどは、SVGをHTML内に直接インライン記述する、または絵文字を利用する。
        
    4.  （今回のカメラ起動のように）ユーザーがその場でアップロードしたファイルをJSでBase64化してメモリ上で扱うのは問題ありません。
        

## 4. ファイル構成の制約（Single-File Mandate）

*   **1ファイル完結が必須:** HTML, CSS, JavaScript はすべて1つの `.html` ファイル（React等の場合は1つの `.jsx` ファイル）に収める必要があります。別ファイルとして `style.css` や `script.js` を作成して読み込むことはできません。
    
*   **ライブラリの読み込み:** Tailwind CSSやThree.jsなどの外部ライブラリは、必ずCDN経由（`<script src="...">`）で読み込む必要があります。
    

## 5. ブラウザAPIのセキュリティ制限

*   **クリップボードAPI (`navigator.clipboard.writeText`) が失敗しやすい:** iframeのセキュリティポリシー（Permissions Policy）により、モダンなクリップボードAPIがブロックされることが多いです。
    
*   **現状のコードへのフィードバック:** 現状の `majinCopy()` 関数内で `document.execCommand('copy')` をフォールバックとして使用しているのは、Canvas環境において**大正解**の対応です。
    
*   **LocalStorage / IndexedDBの揮発性:** Canvasの再レンダリングやセッション切り替え時に、ブラウザのローカルストレージがクリアされる、あるいはアクセスできなくなることがあります。データの永続化には、現在実装されているような外部データベース（GAS等）の利用が必須です。
    

## 6. UIのレスポンシブ対応

*   Canvas環境のウィンドウ幅はユーザーが自由にリサイズできるため、固定ピクセル（`width: 800px` など）でのレイアウトはレイアウト崩れの原因になります。
    
*   Tailwindのユーティリティクラス（`w-full`, `max-w-2xl`, `%`, `vw` など）を活用し、流動的なレイアウト（Fluid Design）にすることが求められます。（現状のアプリはこの点もクリアできています）
    

**【総括と次のステップの提案】** 現在の「料理レシピ・食材管理アプリ」は、GASを用いた通信回避やクリップボードの対応など、Canvasの制限を非常によく理解した上で作られた素晴らしいコードです。 唯一の懸念点は **`alert()` と `confirm()` の使用** です。もしご希望であれば、これらを安全なカスタムUI（Toastや独自確認モーダル）に置き換えるコードの修正を行いましょうか？
