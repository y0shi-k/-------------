1. まず「ハーネス」とは何か

ハーネス = AIモデル以外の、AIをうまく動かすための外側の仕組みです。

文書では、AIエージェントを次のように整理しています。

Agent = Model + Harness

つまり、

Model：Claude / GPT / Codex などのAI本体
Harness：AIに何を見せるか、何を使わせるか、どう検証するか、どう制限するか、どう記録するか

です。

非エンジニア向けに言うと、

モデル = 作業者本人
ハーネス = 作業指示書、チェックリスト、道具箱、上司、品質検査、作業ログ

です。

AIの出力が「日によってブレる」「完了と言うが壊れている」「途中で文脈を忘れる」という問題は、モデルの頭の良し悪しだけではなく、作業環境の設計不足として捉えるべき、というのが全体の主張です。

2. ハーネスが担う5つの役割

ハーネスの役割は大きく5つです。

コンテキスト管理
AIに何を見せるかを決める
例：仕様書、過去の決定、コーディングルール、デザイン方針
ツールアクセス
AIに何を使わせるかを決める
例：ファイル編集、Web検索、テスト実行、Git操作
フィードバックループ
AIの成果物を検証し、修正させる
例：テスト、レビュー、リンター、AIレビューア
ガードレール
AIにやってはいけないことを制限する
例：特定フォルダ以外を書き換えない、本番DBを触らない、勝手に削除しない
可観測性
AIが何をしたか後から見えるようにする
例：ツール使用履歴、コスト、変更ログ、失敗ログ

重要なのは、プロンプトを上手く書くことだけがハーネスではないという点です。むしろ、プロンプトはハーネスの一部にすぎません。

3. 作るべきハーネスの全体像

文書群を統合すると、ハーネスは次の3層で考えるのが実務向きです。

A. ミクロハーネス

個別タスクをうまくやらせる仕組みです。

例：

CLAUDE.md
Skills
Hooks
サブエージェント
テスト
リンター
レビュー用プロンプト
作業ディレクトリ制限

これは「どう作るか」の層です。文書では、ミクロハーネスは実装・テスト・レビュー・ドキュメントなどを分担させる開発チームのようなものとして整理されています。

B. マクロハーネス

何を・なぜ・いつやるかを決める仕組みです。

例：

今週やるタスクの優先順位
KPIや目的との紐づけ
どこまでできたら完了か
失敗時の再計画
定期実行の設計

文書では、サブエージェントを増やすだけでは継続開発にならず、上流の意思決定と状態管理がボトルネックになると説明されています。

C. Project OS

状態・決定・依存関係を残す場所です。

例：

最新仕様
決定事項
却下した案
未解決課題
作業履歴
次にやること
ナレッジ

文書では、会話ログを主役にすると「どれが最新の決定か分からない」問題が起きるため、主役は会話ではなく構造化された状態にすべきだとされています。

4. 最小構成で作るならこれ

非エンジニアがClaude Code / Codexを使って実務に導入するなら、最初から大規模な自作エージェントシステムは不要です。

まずはこの構成で十分です。

project/
├── CLAUDE.md
├── docs/
│   ├── project-brief.md
│   ├── rules.md
│   ├── decisions.md
│   └── review-criteria.md
├── tasks/
│   ├── backlog.md
│   └── current-task.md
├── .claude/
│   ├── skills/
│   └── agents/
└── scripts/
    ├── check.sh
    └── test.sh

それぞれの役割はこうです。

要素	役割	非エンジニア向けの意味
CLAUDE.md	AIへの基本ルール	作業者向けの社内ルール
project-brief.md	プロジェクト概要	何を作っているか
rules.md	守るべきルール	開発・デザイン・運用の制約
decisions.md	決定事項	「なぜそうしたか」の記録
review-criteria.md	評価基準	完了判定チェックリスト
current-task.md	今やる作業	作業指示書
skills	定型作業	手順書・テンプレ
agents	役割別AI	担当者の分担
check.sh / test.sh	自動検証	機械的な検査
5. 作り方：ステップ順
Step 1. まずCLAUDE.mdを作る

CLAUDE.mdとは、Claude Codeに読ませるプロジェクトの基本指示書です。

文書では、CLAUDE.mdは外部ハーネスの基本的な入口であり、長大な説明書ではなく、ルールへのポインタとして機能させるのがよいとされています。

最初の例：

# Project Instructions

## Goal
このプロジェクトは、○○を実現するためのWebアプリです。

## Important Rules
- 作業前に docs/project-brief.md を読む
- 仕様変更があれば docs/decisions.md に記録する
- 実装後は必ずテストまたは動作確認を行う
- 推測で大きな変更をしない
- 既存UIを壊さない
- 変更内容を最後に要約する

## References
- 仕様: docs/project-brief.md
- ルール: docs/rules.md
- 完了条件: docs/review-criteria.md
- 現在の作業: tasks/current-task.md

ポイントは、CLAUDE.mdに全部を書かないことです。
全部詰め込むとAIが重要な部分を見失いやすくなります。

Step 2. 「本質ドキュメント」を作る

文書では、ハーネスの核として 本質ドキュメント が出てきます。

本質ドキュメントとは、誰が見ても正しい原理原則をまとめた評価基準です。

例えば、UI改善なら：

# UI設計の本質ドキュメント

## 原則
- ユーザーが次に何をすべきか一目で分かること
- 重要な情報ほど視線が先に向くこと
- 同じ意味の操作は同じ見た目にすること
- 余白・色・文字サイズに一貫性を持たせること
- 見た目の装飾より、操作の迷いを減らすことを優先する

## やってはいけないこと
- 色だけで重要度を表現する
- 画面ごとにボタンの位置や意味を変える
- 1画面に判断材料を詰め込みすぎる

開発なら：

# 開発の本質ドキュメント

## 原則
- 変更範囲を小さくする
- 既存機能を壊さない
- 実装後に確認可能な状態にする
- 仕様と実装の差分を記録する
- 複雑な処理は分割する

## やってはいけないこと
- 動作確認なしで完了扱いにする
- 不要な大規模リファクタリングを混ぜる
- エラーを握りつぶす

文書では、本質ドキュメントはプロジェクト固有ルールと混ぜず、領域ごとに育てるべきだとされています。

Step 3. レビューアパターンを作る

レビューアパターンとは、AIに作業させたあと、別のAIに評価させる仕組みです。

文書では、ハーネスの核は 評価者 + 評価基準 のペアだとされています。

構造はこうです。

実装AI
  ↓
成果物
  ↓
レビューAI
  ↓
フィードバック
  ↓
実装AIが修正

大事なのは、単に「レビューして」と言わないことです。

悪い例：

このコードをレビューして

良い例：

docs/review-criteria.md と docs/essence-development.md を基準にレビューしてください。
以下の観点で、重大度つきで指摘してください。

1. 仕様とズレていないか
2. 既存機能を壊していないか
3. 不要な大規模変更がないか
4. テスト・確認方法があるか
5. 非エンジニアが見ても変更内容を理解できるか
Step 4. ガイドとセンサーを分けて設計する

文書では、ハーネスを ガイド と センサー に分けて考える整理が紹介されています。

種類	意味	例
ガイド	作業前にミスを防ぐもの	CLAUDE.md、ルール、スキル、手順書
センサー	作業後にミスを見つけるもの	テスト、リンター、レビュー、チェックリスト

非エンジニア向けに言うと、

ガイド = 作業前の指示書
センサー = 作業後の検査

です。

どちらか片方だけでは不十分です。
文書でも、ガイドだけでは効いているか分からず、センサーだけでは同じミスを繰り返すため、両方をセットで設計すべきとされています。

最初に作るなら、順番はこれです。

センサー：テスト・チェックコマンド
ガイド：CLAUDE.md・rules.md
レビューア：AIレビュー
記録：decisions.md・作業ログ

文書でも、まずはリンターやテストなどの計算的センサーから始めるのが推奨されています。

6. スキルの作り方

文書では、スキルを4種類に分類しています。

スキル種類	意味	例
辞書型	知識を渡す	API仕様、デザインルール、業務知識
手順型	作業手順を渡す	レビュー手順、デプロイ手順
生成手順型	ツール込みの作業	コード生成、ファイル変換
アイデンティティ型	判断基準・美学を渡す	UIの好み、文章トーン、設計思想

非エンジニアが最初に作るべきは、手順型スキルと辞書型スキルです。

手順型スキルの例：画面改善レビュー
# ui-review-skill

## Input
- 対象画面
- 変更目的
- 既存スクリーンショット
- ユーザーが困っている点

## Steps
1. 画面の目的を確認する
2. ユーザーが最初に見るべき情報を特定する
3. 迷いやすい操作を洗い出す
4. 余白・文字サイズ・色・ボタン配置を確認する
5. 改善案を優先度つきで出す

## Output
- 問題点
- 改善案
- 優先度
- 実装指示
辞書型スキルの例：プロジェクト用語集
# project-domain-knowledge

## 用語
- 在庫：ユーザーが現在持っている食材
- レシピ候補：在庫から作れる料理案
- 買い足し：不足している食材
- 消費期限：食材を使い切る目安

## 判断基準
- 食材登録は手入力より写真認識を優先する
- 料理提案では、買い足しが少ない案を優先する
- ユーザーに難しい設定を要求しない
7. サブエージェントの作り方

サブエージェントとは、役割を限定したAI担当者です。

文書では、1つのエージェントに多くを任せると品質が落ちるため、関心事を分離すべきとされています。

例えば、アプリ開発なら：

親エージェント
├── 要件整理エージェント
├── UIレビューエージェント
├── 実装エージェント
├── テストエージェント
└── ドキュメント更新エージェント

ただし注意点があります。

サブエージェントを増やせば良いわけではありません。

文書でも、重要なのは人格を増やすことではなく、責務と状態を分けることだとされています。

最初はこの3つで十分です。

実装担当
コードを書く
レビュー担当
問題を見つける
記録担当
決定事項・次アクションを残す
8. Hooksの考え方

Hooksとは、AIの作業前後に自動で走らせる処理です。

非エンジニア向けに言うと、
**「AIが何かしたら、自動でチェックや記録を入れる仕組み」**です。

例：

ファイル編集後にテストを走らせる
作業完了時に変更内容をログに残す
危険なコマンド実行前に止める
コストやツール使用回数を記録する

文書では、Agent Coreの役割として、Hooks処理、ツールコール履歴、コスト情報の取得・記録が挙げられています。

最初に作るなら、Hooksはこの3つでよいです。

Hook	目的
作業開始時	current-task.mdを読む
ファイル編集後	test/checkを実行
作業終了時	decisions.md / work-log.md に記録
9. ナレッジ管理の作り方

文書では、継続運用にはナレッジ管理が重要とされています。

最初はDBを使わず、Markdownファイルで十分です。

docs/
├── decisions.md       # 決定事項
├── learnings.md       # 学び・失敗パターン
├── rejected-ideas.md  # 却下した案
├── glossary.md        # 用語集
└── todo.md            # 未完了タスク

特に重要なのは decisions.md です。

例：

# Decisions

## 2026-05-30
### 決定
食材登録画面では、写真認識を主導線にする。

### 理由
手入力中心だとユーザーの負担が大きく、継続利用されにくい。

### 却下した案
初回から詳細なカテゴリ選択をさせる案。

### 次に確認すること
写真認識の誤判定時に、ユーザーが簡単に修正できるUI。

AIに毎回このファイルを読ませることで、同じ議論のやり直しを減らせます。

10. 非エンジニア向けの実装ロードマップ
Phase 1：まず1日で作る最小ハーネス

やること：

CLAUDE.mdを作る
docs/project-brief.mdを作る
docs/rules.mdを作る
docs/review-criteria.mdを作る
tasks/current-task.mdを作る

目的：

AIが毎回迷わない
作業指示が安定する
完了条件が明確になる

この段階では、Hooksやサブエージェントは不要です。

Phase 2：レビューアを入れる

やること：

実装後に必ずAIレビューを挟む
review-criteria.mdを基準に評価させる
指摘を「重大・中・軽微」に分ける

目的：

「完了しました」を鵜呑みにしない
品質を客観的に確認する
Phase 3：記録を残す

やること：

decisions.mdを更新させる
work-log.mdを更新させる
次回作業をcurrent-task.mdに残す

目的：

会話ログ依存を減らす
次回再開しやすくする
Phase 4：スキル化する

繰り返す作業をスキル化します。

候補：

UIレビュー
コードレビュー
仕様整理
リリース前チェック
Gitコミット前チェック
スクショ比較
不具合調査

目的：

毎回同じ説明をしなくてよくする
作業品質を安定させる
Phase 5：サブエージェント化する

役割が明確になってきたら分けます。

候補：

UI担当
実装担当
テスト担当
レビュー担当
仕様整理担当

目的：

1つのAIに全部抱えさせない
作業の抜け漏れを減らす
11. そのまま使えるプロンプト
ハーネス初期構築プロンプト
このプロジェクトにAIエージェント用の最小ハーネスを作ってください。

目的は、Claude Code / Codexでの作業品質を安定させることです。

以下のファイルを作成してください。

1. CLAUDE.md
2. docs/project-brief.md
3. docs/rules.md
4. docs/review-criteria.md
5. docs/decisions.md
6. tasks/current-task.md

要件：
- 非エンジニアでも読める日本語で書く
- CLAUDE.mdには全部を書かず、各docsへの参照を中心にする
- review-criteria.mdには、作業完了判定のチェック項目を入れる
- decisions.mdには、決定・理由・却下案・次に確認することのフォーマットを用意する
- current-task.mdには、目的・作業範囲・完了条件・禁止事項のフォーマットを用意する
作業開始プロンプト
作業前に以下を読んでください。

- CLAUDE.md
- docs/project-brief.md
- docs/rules.md
- docs/decisions.md
- tasks/current-task.md

読んだ後、以下を確認してください。

1. 今回の作業目的
2. 変更してよい範囲
3. 変更してはいけない範囲
4. 完了条件
5. 不明点・リスク

確認後、実装計画を短く出してください。
作業後レビュー依頼プロンプト
今回の変更をレビューしてください。

参照する基準：
- docs/review-criteria.md
- docs/rules.md
- docs/project-brief.md

レビュー観点：
1. 目的に合っているか
2. 既存機能を壊していないか
3. 不要な変更が混ざっていないか
4. UI/UX上の違和感がないか
5. テストまたは確認方法があるか
6. 次回に残すべき注意点があるか

出力形式：
- 重大な問題
- 修正推奨
- 軽微な改善
- 問題なし
- 次にやること
作業終了時プロンプト
作業終了処理をしてください。

以下を更新してください。

1. docs/decisions.md
   - 今回決めたこと
   - 理由
   - 却下した案
   - 次に確認すること

2. docs/work-log.md
   - 実施内容
   - 変更ファイル
   - 確認結果
   - 未解決事項

3. tasks/current-task.md
   - 完了したこと
   - 次にやること
   - 次回作業者への注意
12. リスク・注意点
リスク1：ルールを書きすぎる

CLAUDE.mdに何でも詰め込むと、AIが重要な指示を見落とします。

対策：

CLAUDE.mdは短く
詳細はdocs配下に分ける
重要ルールは10個以内にする
リスク2：AIレビューを信用しすぎる

AIレビューアも間違えます。

対策：

重要な判断は人間が見る
テスト・リンターなど機械的チェックを優先する
AIレビューは補助と考える

文書でも、AIがもっともらしい誤情報を生成する可能性があるため、本質ドキュメントへの取り込みは人間が判断すべきとされています。

リスク3：サブエージェントを増やしすぎる

役割分担が曖昧なまま増やすと、かえって混乱します。

対策：

最初は「実装」「レビュー」「記録」の3つで十分
役割・入力・出力を明確にする
同じ情報を全員に渡しすぎない
リスク4：会話ログに依存する

チャットだけで進めると、後で最新状態が分からなくなります。

対策：

decisions.mdに決定を残す
current-task.mdを更新する
作業ログを残す
13. あなたの用途に合わせた最適解

あなたの使い方、つまり 非エンジニアがClaude Code / Codexでバイブコーディングする 前提なら、最初に目指すべきはこれです。

大規模な自作エージェント基盤
ではなく、
プロジェクトごとの軽量ハーネス

優先順位は以下です。

CLAUDE.md
docs/project-brief.md
docs/review-criteria.md
docs/decisions.md
作業後レビュー
作業終了ログ
繰り返し作業のスキル化
必要になったらサブエージェント化

最初からSlack Bot、Agent Core、DB、ダッシュボード、Cronなどを作る必要はありません。文書内にはSlackボット、Agent Core、Supabase/PostgreSQL、ダッシュボード、オーケストレーターなどの構成例もありますが、これはかなり高度な運用です。

今の段階では、Markdownファイル中心のハーネスで十分効果が出るはずです。

14. 最終まとめ

ハーネス作りは、次の一文に圧縮できます。

AIに「頑張って良い成果を出して」と頼むのではなく、良い成果が出やすく、悪い成果が検出され、判断が記録される作業環境を作ること。

実務での最小セットはこれです。

CLAUDE.md        = AIへの基本ルール
project-brief   = 何を作るか
rules           = 守ること
review-criteria = 完了条件・品質基準
decisions       = 決定の記録
current-task    = 今やること
reviewer        = AIレビュー係

この構成を入れるだけで、Claude Code / Codexの「日によってブレる」「前回の話を忘れる」「完了したと言うが壊れている」をかなり減らせます。
