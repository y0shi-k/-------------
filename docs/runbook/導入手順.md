# 導入手順

## 目的

この手順は、このプロジェクトで AI ハーネスを使って開発を始める前に、何を確認・埋めてから実装を始めるかを示します。

## 最初に埋めるもの

1. `AGENTS.md`
   - プロジェクト名: Stock Master（料理レシピ・食材管理アプリ）
   - 正本: `app.html`, `要件定義書.md`, `AGENTS.md`, `.agents/`, `harness/*.json`
   - 生成物: GASデプロイ後のWebアプリ、Spreadsheet内データ、Drive保存画像
   - verify: `python3 -c "import html.parser; html.parser.HTMLParser().feed(open('app.html').read())" && grep -q 'executeGAS' app.html && grep -q 'GAS_URL' app.html && echo 'VERIFY_PASSED'`
   - 任意監査: 現時点では未定（Phase2以降で必要に応じて設定）
   - 安全データ: GAS_URL、Gemini APIキー、SS_ID_RECIPE_APP、ユーザー食材データ
   - 実行境界: GASデプロイ、Spreadsheet手動編集、Driveアップロードは明示依頼のみ
2. `harness/registry.json`
   - verify 対象: HTML構文、executeGAS存在、GAS_URL存在
   - manual smoke 項目: HTML構文チェック、GAS通信パターン、スキーマ整合性、UI表示整合性
3. `harness/change_evals.json`
   - 重要機能ごとの eval: schema_change, gas_pattern_change, ui_component_addition, phase_transition
4. `harness/phase_gates.json`
   - 完了判定の gate: spec_ready, implementation_ready, verify_passed, manual_smokes_done, review_ready, report_ready

## 実装開始前の順番

1. `project-os/specs/` に最初の `SPEC` を作る
2. `project-os/tickets/` に対応する `TKT` を作る
3. `implementation_ready` になるまで required_evals と changed_paths を埋める
4. verify コマンドが実行できることを確認する
5. その後に実装を開始する

> 完了した `TKT` / `SPEC` は `/finalize` が `project-os/tickets/completed/` `project-os/specs/completed/` へ自動移動する（ファイル名・`id` は不変）。現役分はトップ階層に残る。

## 人が見るポイント

- 正本と生成物の境界が埋まっているか
- verify コマンドが具体化されているか
- manual smoke 項目がこのアプリの重要操作に合っているか
- `project-os/artifacts/TKT-xxxx/` に結果を残す前提ができているか
