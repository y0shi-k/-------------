# 公開前セキュリティ確認

## 結論

初期公開では、誰でも新規登録できる状態にしません。ユーザーはSupabase Dashboardの管理者作成、または招待運用で追加します。

この手順は、`supabase/config.toml` と `web/next.config.ts` のコード側設定に加えて、本番Supabase/Vercelで人が確認する項目です。

## Supabase Auth

Supabase Dashboardで次を確認します。

```text
Authentication
→ Providers
→ Email
```

確認すること:

- Email signupを無効にする
- メール確認を有効にする
- パスワード最小文字数を8文字以上にする
- パスワード要件を英字・数字必須にする

一般公開で自由登録に戻す場合は、先に次を検討します。

- メール確認を有効にしたままにする
- CAPTCHAを有効にする
- 不審な登録やログイン失敗が増えていないか確認する

## ユーザー追加

初期公開では、Supabase Dashboardから管理者がユーザーを作ります。

```text
Authentication
→ Users
→ Add user
```

注意:

- 仮パスワードをチャットやMarkdownに書かない
- 本人に安全な別経路で共有する
- 不要になったテストユーザーは削除する

## Vercel環境変数

VercelのProject Settingsで次を確認します。

```text
Settings
→ Environment Variables
```

確認すること:

- `NEXT_PUBLIC_SUPABASE_URL` は本番SupabaseのURL
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` はanon public key
- `SUPABASE_SERVICE_ROLE_KEY` は必要な場合だけ設定し、ブラウザへ出さない
- `GEMINI_API_KEY` は実値をコードやMarkdownに書かない

`SUPABASE_SERVICE_ROLE_KEY` や `GEMINI_API_KEY` が漏れると、データ読み書きやAI API利用につながる可能性があります。

## セキュリティヘッダー

`web/next.config.ts` で次を返します。

- `Content-Security-Policy`
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`

CSPは、ブラウザが読み込んでよい接続先を制限する仕組みです。強くしすぎると、Supabase通信、署名付き写真URL、開発中の画面更新が壊れるため、初回は保守的に設定しています。

## 手動確認

公開前に次を確認します。

- 未ログインで通常画面へアクセスすると `/login` に移動する
- 既存ユーザーでログインできる
- 写真付き料理履歴が表示できる
- 食材写真のアップロードとAI解析を実行できる
- レシピAI生成を実行できる
- スマホ幅でログイン画面と主要画面が崩れていない

## 問題が出た場合

### ログインできない

原因:

新規登録停止やメール確認設定により、未確認ユーザーや未作成ユーザーでログインしている可能性があります。

影響:

ユーザーがアプリに入れません。

修正方法:

Supabase Dashboardで対象ユーザーが存在し、メール確認済みか確認します。

### 写真が表示されない

原因:

CSPの `img-src` が写真URLを許可していない可能性があります。

影響:

料理履歴や写真プレビューが空になります。

修正方法:

ブラウザの開発者ツールでCSPエラーのURLを確認し、必要な接続先だけ `web/next.config.ts` に追加します。

### Supabase通信が失敗する

原因:

CSPの `connect-src` がSupabase URLを許可していない可能性があります。

影響:

ログイン、保存、削除、AI API呼び出し前の認証確認が失敗します。

修正方法:

本番SupabaseのURLが `https://*.supabase.co` に該当するか確認します。独自ドメインを使う場合は、そのドメインだけを `connect-src` に追加します。
