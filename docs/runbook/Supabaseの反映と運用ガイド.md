# Supabase の反映と運用ガイド（非エンジニア向け・完全版）

> このドキュメントは、Stock Master Web版で使っている **Supabase** について、
> 「変更をどうやって反映するのか」「これから使い続けるうえで知っておくべきこと」を、
> 専門用語をできるだけかみ砕いて、省略せずに説明します。
>
> - 最初の登録（アカウント作成・プロジェクト作成・`.env.local` 設定）は
>   `docs/runbook/Supabase登録の詳細手順.md` を参照してください。
> - このガイドは、その後の **日常運用**（変更の反映・確認・安全管理）を扱います。

---

## 0. まず結論（いちばん大事なことだけ）

1. **データベースの設計図は「コード（migration ファイル）」が正本**です。
   Supabase の管理画面（ダッシュボード）で手作業でテーブルをいじらないでください。
2. 変更を Supabase に反映する標準の流れは、リポジトリの `supabase/migrations/` にある
   SQL ファイルを **`supabase db push`** で DB に流すことです。
3. **本番（や hosted）DB への反映は「外向きの危険操作」**です。
   AI（Claude/Codex）には勝手に実行させず、**あなたが明示的に承認・実行**します。
4. **秘密のカギ（特に `service_role` key と DB パスワード）は絶対に公開しない**。
   チャット・Git・Markdown・スクショに貼らない。

この4つを守れば、大きな事故はほぼ防げます。以下はその理由と具体手順です。

---

## 1. Supabase はこのアプリで何をしているのか

Supabase は「Webアプリの裏側」をまとめて引き受けてくれるサービスです。
Stock Master Web版では、Canvas版で使っていた Google スプレッドシート／ドライブ／GAS の
**代わり**に Supabase を使っています。役割は大きく3つです。

| 役割 | 何をするか | このアプリでの例 |
|---|---|---|
| **データベース (Database)** | 食材・レシピ・献立などのデータを表（テーブル）で保存する | `inventory_items`（在庫）, `recipes`（レシピ）など |
| **認証 (Auth)** | ログイン・ユーザー管理 | メール＋パスワードでログイン |
| **ストレージ (Storage)** | 写真などのファイルを保存する | 料理写真・食材写真（`photos` バケット） |

つまり「データの保管庫＋ログイン＋写真置き場」を1つで担当しています。

## 2. 専門用語をやさしく（用語集）

はじめての人がつまずきやすい言葉を、日常のたとえで説明します。

| 用語 | やさしい説明 | たとえ |
|---|---|---|
| **プロジェクト (Project)** | Supabase 上のアプリ1個分の入れ物 | 「お店1店舗」 |
| **テーブル (Table)** | データを入れる表 | 「Excelのシート1枚」 |
| **行 (Row) / 列 (Column)** | 表の1件 / 表の項目名 | 「1人分のデータ / 名前・数量などの欄」 |
| **マイグレーション (migration)** | DBの設計変更を1ファイルに書いた「変更指示書」 | 「お店の改装図面」 |
| **スキーマ (schema)** | DB全体の設計（どんな表・列があるか） | 「お店全体の間取り」 |
| **RLS（行レベルセキュリティ）** | 「自分のデータしか見えない／触れない」ルール | 「自分のロッカーしか開かない鍵」 |
| **バケット (bucket)** | Storage の中のフォルダの大箱 | 「写真用の倉庫」 |
| **環境変数 (.env)** | アプリに渡す設定値（URLやカギ） | 「お店の鍵と住所を書いたメモ」 |
| **anon key（公開キー）** | ブラウザから繋ぐ用の、見えてもよいカギ | 「店の入口の番号札」 |
| **service_role key（秘密キー）** | サーバー専用の超強力なカギ。**絶対公開しない** | 「店長のマスターキー」 |
| **アクセストークン (access token)** | ログイン中の本人を表す一時的な合言葉 | 「入館中だけ有効な入館証」 |
| **CLI（コマンドラインツール）** | `supabase ...` と打って操作する道具 | 「リモコン」 |
| **Studio（スタジオ）** | Supabase の管理画面（ブラウザ） | 「店の管理デスク」 |

## 3. 「環境」の考え方 ― ローカル / hosted / 本番

データベースは **同じ設計でも“置き場所”が複数**あります。これを「環境」と呼びます。

| 環境 | 何のため | このプロジェクトの状況 |
|---|---|---|
| **ローカル (local)** | 自分のPCの中だけで動かす練習用DB | **Docker が入っていないため、現状は使えない**（`supabase start` 不可） |
| **hosted（検証用クラウド）** | クラウド上の確認用DB | プロジェクト参照ID `wwtompvneobysieofxkl` を使用。危険変更はここで実検証している |
| **本番 (production)** | 実際の利用者が使うDB | 公開段階で運用（今は限定運用・新規登録は停止中） |

### このプロジェクト特有の事情（重要）

- 通常は「ローカルDBで試す → 問題なければ本番へ」が理想ですが、
  **このPCには Docker が無いため、本物のローカルSupabaseを起動できません**。
- そのため、これまでの危険変更（例: お気に入り `is_favorite` の追加）は、
  **あなたの承認のうえで hosted（`wwtompvneobysieofxkl`）に `supabase db push` して実検証**してきました。
  （経緯は `project-os/knowledge/decisions.md` 2026-06-04 に記録）

### `.env` ファイルの使い分け

`web/` フォルダには接続先を切り替えるための環境変数ファイルが複数あります。

| ファイル | 役割 | Git管理 |
|---|---|---|
| `web/.env.example` | 記入見本（カラの雛形） | **される**（中身は空） |
| `web/.env.local` | 実際に使う値（既定の接続先） | されない（秘密） |
| `web/.env.dev.local` | ローカル/開発向けの値 | されない（秘密） |
| `web/.env.hosted.local` | hosted（クラウド検証）向けの値 | されない（秘密） |

> `.local` が付くファイルは **秘密の実値が入る** ため、Git に入りません（`.gitignore` で除外）。
> 中身（カギ）を人に見せない・貼らないでください。

主な環境変数（名前だけ。値は秘密）:

| 変数名 | 用途 | 公開してよい？ |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | プロジェクトの住所(URL) | ◯（ブラウザに出てよい） |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | ブラウザ接続用の公開キー | ◯ |
| `SUPABASE_SERVICE_ROLE_KEY` | サーバー専用の超強力キー | ✕ 絶対ダメ |
| `SUPABASE_DB_PASSWORD` | DBのパスワード | ✕ 絶対ダメ |
| `GEMINI_API_KEY` | AI(Gemini)用のキー | ✕ |

> 覚え方:「`NEXT_PUBLIC_` が付くものは“公開してよい”、付かないものは“秘密”」。
> `service_role` に `NEXT_PUBLIC_` を**絶対に付けない**でください（付けると世界中に鍵が見えます）。

## 4. 「変更を反映する」とは何か（一番大事な考え方）

このプロジェクトでは、DBの設計は **コードが正本**です。

```
正本（設計図）          反映先（実物のDB）
supabase/migrations/  ──[ supabase db push ]──▶  hosted / 本番 のデータベース
  *.sql（変更指示書）
```

- 「列を増やす」「テーブルを作る」などの変更は、まず `supabase/migrations/` に
  **SQLファイル（migration）として書いて Git に残します**。
- それを `supabase db push` で実際のDBへ流すと、DBがその設計に追いつきます。

### なぜダッシュボードで手作業しないのか

- 画面で手作業すると、**コード（正本）と実物のDBがズレ**ます。
- ズレると「なぜこの列があるんだっけ？」が後から分からなくなり、別環境で再現もできません。
- migration に残せば、**履歴・理由・順番**がすべて Git に残り、誰でも同じ状態を再現できます。

### migration ファイルの名前のルール

```
supabase/migrations/20260605120000_user_image_columns.sql
                    └─ 日時(YYYYMMDDHHMMSS) ┘ └─ 内容がわかる名前 ┘
```

- 先頭の数字（日時）で **適用される順番**が決まります。新しい変更ほど大きい数字。
- 1ファイル＝1まとまりの変更。過去のファイルは**書き換えず**、新しいファイルを足していきます。

## 5. 反映の手順（ステップ・バイ・ステップ）

> ⚠️ **これは外向きの危険操作です。** 本番/hosted への反映は、内容を理解し、
> バックアップ方針を確認したうえで、**あなた自身が**実行してください。
> AIは差分の作成と確認まで。`push` はあなたの承認・実行が前提です。

### 前提（最初の1回だけ）

1. **Supabase CLI を入れる**（道具のインストール）。
   Mac なら例: `brew install supabase/tap/supabase`（公式の最新手順に従ってください）。
2. **ログインする**（CLIに自分のSupabaseアカウントを教える）。
   ログインはブラウザが開く対話式なので、このチャットからではなく
   **あなたのターミナルで** 実行してください。Claude Code を使っているなら、
   入力欄に次のように `!` を付けて打つと、その場で実行できます。

   ```
   ! supabase login
   ```

3. **プロジェクトを紐づける（link）**。どのクラウドDBに反映するかを指定します。

   ```
   ! supabase link --project-ref wwtompvneobysieofxkl
   ```

   （`wwtompvneobysieofxkl` は現在の hosted 検証用プロジェクトの参照ID。
   本番に別プロジェクトを使う場合は、その参照IDに置き換えます。）

### 反映の本番手順

1. **今どこまで適用済みか確認する**

   ```
   ! supabase migration list
   ```

   ローカルにあるmigrationと、クラウドDBに適用済みのものの差が分かります。

2. **「これから何が起きるか」を先に見る（空打ち＝dry run）**

   ```
   ! supabase db push --dry-run
   ```

   実際には変更せず、**何のSQLが流れる予定か**だけ表示します。ここで内容を必ず目視確認。

3. **実際に反映する**

   ```
   ! supabase db push
   ```

   未適用のmigrationが順番にDBへ流れます。

4. **反映後に確認する**
   - Supabase の **Studio（管理画面）→ Table Editor** で、狙った列・テーブルが増えているか見る。
   - **既存データが消えていないか**（件数が変わっていないか）を見る。
   - アプリ（`npm run dev` または公開URL）を開いて、画面が普通に動くか確認する。

### うまくいかない・間違えたときの考え方

- migration は「足し算で前に進む」のが基本です。**一度流したものを“なかったこと”にするのは難しい**ので、
  間違いを直すときは「**打ち消す内容の新しいmigrationを足す**」のが安全です。
- 削除・上書き・列の型変更など **取り返しのつきにくい操作**は、特に慎重に。
  事前に Supabase ダッシュボードの **バックアップ**機能や、重要データのエクスポートを検討してください。
- 不安なときは、まず hosted（検証用）で試してから本番に同じmigrationを流します。

## 6. データを安全に保つ仕組み（RLS と Storage）

このアプリは「**自分のデータしか見えない・触れない**」ように作られています。仕組みは2つ。

### RLS（行レベルセキュリティ）

- すべてのユーザーデータの表には `user_id`（持ち主）列があります。
- RLS のルールは `auth.uid() = user_id`、つまり
  **「ログイン中の本人＝その行の持ち主」のときだけ読み書きできる**、という意味です。
- だから、ある人のレシピが他人に見えることはありません。

### Storage（写真）

- 写真は **非公開バケット `photos`** に入ります（誰でも見られる公開状態ではない）。
- 保存場所（パス）は必ず `<自分のuser_id>/...` で始まり、
  ルールにより **本人以外はその写真を読めません**。
- 表示するときは「**短時間だけ有効な署名付きURL**」を発行します
  （リンクを知っても時間が経つと無効になる、使い捨ての見学パスのようなもの）。

> ポイント: **公開URLを保存しない**。DBには「写真のしまい場所（パス）」だけ保存し、
> 見せるときに毎回その場で短命リンクを作る、という方針です。

## 7. やってはいけないこと / 守ること（安全ルール）

- ❌ `service_role` key・DBパスワードを **チャット/Git/Markdown/スクショ**に出す。
- ❌ `SUPABASE_SERVICE_ROLE_KEY` に `NEXT_PUBLIC_` を付ける（ブラウザに漏れる）。
- ❌ ダッシュボードで手作業でテーブルを変える（コードとズレる）。
- ❌ AIに本番/hosted への `db push` や削除を勝手にやらせる。
- ✅ DB変更は migration ファイルで残し、Gitにコミットする。
- ✅ 本番/hosted への反映は **あなたが内容を確認してから**実行する。
- ✅ 反映後は「狙った変更が入ったか」「既存データが消えてないか」を必ず確認する。
- ✅ 秘密の値は `.env.local`（Git管理外）にだけ書く。

## 8. シードデータ（デモ用データ投入）について

`scripts/seed-demo-recipes.mjs` は、デモ用のレシピ・食材を**追加するだけ**のスクリプトです
（TKT-0172）。次の点を理解しておいてください。

- **自動実行しません。** あなたが必要なときだけ手で実行します。
- 実行には、ログイン中の本人を表す一時的な **アクセストークン**（`SUPABASE_AUTH_TOKEN`）を使います。
  これも秘密情報なので、貼り付け・共有は避けてください。
- 既存の同名データは**上書きせずスキップ**します（追加のみ。あなたの既存データを壊しません）。

実行イメージ（値は自分のものに置き換え。トークンは秘密）:

```bash
NEXT_PUBLIC_SUPABASE_URL=... \
NEXT_PUBLIC_SUPABASE_ANON_KEY=... \
SUPABASE_AUTH_TOKEN=... \
node scripts/seed-demo-recipes.mjs
```

## 9. よくあるトラブルと対処

| 症状 | よくある原因 | 対処 |
|---|---|---|
| `supabase db push` で「ログインして」と言われる | CLIにログインしていない／linkしていない | `supabase login` → `supabase link --project-ref ...` |
| 画面に「ログインが必要」と出てデータが見えない | RLSは正常。未ログイン or 別ユーザー | 正しい本人でログインし直す |
| 写真が表示されない | 署名付きURLの期限切れ／パスが本人配下でない | 画面を開き直す（再発行）／パス先頭が自分のuser_idか確認 |
| 反映したのに列が増えていない | migrationを足し忘れ／push先の環境を間違えた | `supabase migration list` で差分確認、push先(link)を確認 |
| 秘密キーを誤って公開してしまった | `service_role`等を貼ってしまった | **すぐにダッシュボードでキーをローテーション(再発行)** し、古い鍵を無効化 |

## 10. このプロジェクト固有のメモ（早見表）

| 項目 | 値・場所 |
|---|---|
| ローカル設定ID (`config.toml`) | `stock-master-web` |
| hosted 検証用プロジェクト参照ID | `wwtompvneobysieofxkl` |
| DB メジャーバージョン | PostgreSQL 17 |
| migration の置き場所（正本） | `supabase/migrations/` |
| 写真バケット | `photos`（非公開） |
| 写真パスの規約 | `<user_id>/<用途>/...`（例: `<uid>/recipe-images/<id>/<uuid>.webp`） |
| RLS の基本ルール | `auth.uid() = user_id` |
| 新規登録 | 既定で停止（`enable_signup = false`／限定運用） |
| DB適用コマンド | `supabase db push`（`--dry-run` で事前確認） |
| Docker | 未導入 → 真のローカルDBは使えず hosted で検証 |

## 11. これだけは、のチェックリスト（反映前）

- [ ] 変更は `supabase/migrations/` に **新しいSQLファイル**として入っているか
- [ ] そのファイルを **Gitにコミット**したか
- [ ] `supabase db push --dry-run` で流れる内容を**目視確認**したか
- [ ] 反映先（link先）が **意図した環境**（hosted か本番か）になっているか
- [ ] 既存データを **壊さない**変更か（削除・上書き・型変更がないか）
- [ ] 反映後に **Studioとアプリで確認**する段取りがあるか
- [ ] 秘密キーをどこにも貼っていないか
