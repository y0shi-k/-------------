# 承認制ログイン移植ガイド（別アプリへの再利用手順）

> Stock Master で構築した「申請＋管理者承認」方式のログインシステム（TKT-0228〜0233 / SPEC-0228）を、
> **別の Next.js + Supabase + Vercel アプリへ移植する**ための手順書。
> このリポジトリのコードが正本（コピー元）。新アプリを作るときにこのファイルを開き、上から順に実施する。

## このシステムの全体像（何が手に入るか）

- **申請＋管理者承認**: 誰でも `/signup` から登録申請できるが、admin が承認するまでアプリ本体に入れない
- **メール確認**: signup 時に確認メール（Supabase 標準メール）
- **承認ゲート**: middleware＋サーバー＋API の3層遮断。未承認/停止ユーザーは `/pending` に固定
- **アプリ内管理画面**: `/admin` で承認/拒否/無効化/再有効化（admin ロール限定）
- **パスワードリセット**: `/forgot-password` → メール → `/reset-password`
- **設計原則**: service role key をアプリに持ち込まない（すべて anon key＋RLS）。
  状態判定は fail-safe（profiles 取得失敗→pending 扱い＝閉じる方向に倒す）

### 技術前提

| 項目 | 値 |
|---|---|
| フレームワーク | Next.js（App Router）+ TypeScript |
| 認証 | Supabase Auth（メール＋パスワード）+ `@supabase/ssr` |
| DB | Supabase Postgres + RLS |
| デプロイ | Vercel（環境変数 `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` のみ） |
| メール | 当面 Supabase 標準（時間あたり数通の制限あり。増えたら SMTP 移行） |

## Step 1: DB 土台（profiles migration）

新アプリの `supabase/migrations/<timestamp>_auth_profiles.sql` として以下をコピーする
（コピー元: `supabase/migrations/20260611120000_auth_profiles.sql`。内容は同一・汎用）。

```sql
create table if not exists public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  email text not null,
  status text not null default 'pending' check (status in ('pending', 'approved', 'disabled')),
  role text not null default 'member' check (role in ('member', 'admin')),
  approved_at timestamptz,
  approved_by uuid references auth.users(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index if not exists profiles_status_idx on public.profiles (status);
create index if not exists profiles_role_idx on public.profiles (role);

-- updated_at 自動更新（新アプリに同等の共通関数が無ければこの定義ごとコピー）
create or replace function public.set_updated_at()
returns trigger language plpgsql as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

drop trigger if exists set_profiles_updated_at on public.profiles;
create trigger set_profiles_updated_at
before update on public.profiles
for each row execute function public.set_updated_at();

-- 新規 auth ユーザー → pending profile を自動作成
create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer
set search_path = public, pg_temp
as $$
begin
  insert into public.profiles (id, email, status, role)
  values (new.id, coalesce(new.email, ''), 'pending', 'member')
  on conflict (id) do nothing;
  return new;
end;
$$;

drop trigger if exists on_auth_user_created on auth.users;
create trigger on_auth_user_created
after insert on auth.users
for each row execute function public.handle_new_user();

-- 既存ユーザーの backfill（締め出し防止: 全員 approved で開始）
insert into public.profiles (id, email, status, role, approved_at)
select u.id, coalesce(u.email, ''), 'approved', 'member', now()
from auth.users u
on conflict (id) do nothing;

-- admin 判定（security definer で RLS 再帰を回避）
create or replace function public.is_admin()
returns boolean
language sql
security definer
stable
set search_path = public, pg_temp
as $$
  select exists (
    select 1 from public.profiles p
    where p.id = auth.uid() and p.role = 'admin'
  );
$$;

alter table public.profiles enable row level security;

drop policy if exists "profiles_select_own" on public.profiles;
drop policy if exists "profiles_select_admin" on public.profiles;
drop policy if exists "profiles_update_admin" on public.profiles;

create policy "profiles_select_own" on public.profiles
  for select using (auth.uid() = id);

create policy "profiles_select_admin" on public.profiles
  for select using (public.is_admin());

create policy "profiles_update_admin" on public.profiles
  for update using (public.is_admin()) with check (public.is_admin());

revoke all on function public.is_admin() from public, anon;
grant execute on function public.is_admin() to authenticated;
```

設計上の要点（変えるときは理由を確認）:
- **insert/delete ポリシーは作らない**（行はトリガーと backfill だけが作る。クライアントから消せない）
- `coalesce(new.email, '')` はトリガー例外で signup 全体が落ちるのを防ぐ null 耐性
- RLS は列単位制御ができないため、admin の update を status 系列に限定するのは**アプリ側の責務**（Step 4）

## Step 3: 認証コアロジック（アプリ非依存・そのままコピー）

以下4ファイルはドメイン非依存の純ロジック＋テスト。**新アプリへ無修正コピーできる**。

| コピー元 | 内容 |
|---|---|
| `web/src/lib/auth/routing.ts` | 4状態ルーティング純関数（unauthenticated/pending/disabled/approved × パス → リダイレクト先） |
| `web/src/lib/auth/account-status.ts` | profiles.status/role の取得（fail-safe: 失敗→pending / member） |
| `web/src/lib/auth/safe-next-path.ts` | メールリンクの `next` サニタイズ（オープンリダイレクト防止） |
| `web/src/lib/auth/pending-content.ts` | /pending の文言出し分け（文言だけ新アプリ向けに調整） |
| `web/src/__tests__/auth-routing.test.ts` ほか auth系テスト | 全分岐の固定（合わせてコピー推奨） |

状態×パスの遷移表（routing.ts の仕様。新アプリで公開パスが増えたら `isPublicPath` に足す）:

| パス \ 状態 | unauthenticated | pending / disabled | approved |
|---|---|---|---|
| `/login` `/signup` `/forgot-password` | 素通し | `/pending` | `/` |
| `/auth/*`（メールリンク処理） | 素通し | 素通し | 素通し |
| `/pending` | `/login` | 素通し | `/` |
| `/reset-password` | `/login` | 素通し（承認ゲートの唯一の例外） | 素通し |
| それ以外（アプリ本体） | `/login` | `/pending` | 素通し |

`web/middleware.ts`（または `web/src/middleware.ts`）の matcher が静的アセット以外を覆っていることを確認する
（API route も承認ゲートの対象にするため）。

## Step 4: 画面・ルート（コピーして見た目だけ調整）

| コピー元 | 新アプリでの調整点 |
|---|---|
| `web/src/app/login/page.tsx` + `components/login-form.tsx` | アプリ名・CSSクラス |
| `web/src/app/signup/page.tsx` + `components/signup-form.tsx` | 同上（承認待ち案内文も） |
| `web/src/app/auth/confirm/route.ts` | **無修正でよい**（verifyOtp type 許可リスト＋next サニタイズ） |
| `web/src/app/pending/page.tsx` | 文言・CSSクラス |
| `web/src/app/forgot-password/` + `reset-password/` 一式 | 文言・CSSクラス |
| `web/src/app/admin/page.tsx` + `components/admin-user-board.tsx` + `lib/auth/admin-profiles.ts` | CSSクラス・ナビ導線 |
| `web/src/app/globals.css` の `auth-*` / `form-*` / `admin-*` クラス | 新アプリのデザインシステムに合わせ書き換え可 |

セキュリティ上**変えてはいけない**実装ポイント:
1. `/admin` の権限判定はサーバーコンポーネントで行う（`fetchAccountRole` → 非 admin は `redirect("/")`）。
   クライアントの isAdmin はナビ表示の出し分けのみに使う
2. admin の update payload は `buildStatusUpdate`（status / approved_at / approved_by のみ）を経由する。
   **role・email を update に含めない**（RLS で列制御できない分の UI 担保）
3. 自分自身の行は操作ボタン非描画＋実行時 early return の二重ガード（admin の自己締め出し防止）
4. forgot-password は未登録メールでも成功と**完全同一の文言**（存在有無の非露出）
5. 生の Supabase エラーメッセージを画面に出さない（日本語変換関数を通す）
6. 新アプリに API route があれば、middleware 任せにせず route 冒頭でも
   `fetchAccountStatus !== "approved"` → 403 を入れる（多層防御。コピー元: `web/src/app/api/ai/recipes/route.ts`）

## Step 5: ローカル Supabase 設定（config.toml）

`supabase/config.toml` の `[auth]` 系をコピー元に合わせる:
- `[auth.email] enable_confirmations = true`（メール確認フローのローカル再現）
- `additional_redirect_urls` に `http://127.0.0.1:3000/auth/confirm`・`http://localhost:3000/auth/confirm`
- `minimum_password_length` は 8 以上
- ローカルの確認メールは Inbucket/Mailpit（`http://127.0.0.1:54324`）で受信確認

## Step 6: 本番適用（順序厳守）

**正本: `docs/runbook/認証本番化の適用手順.md`**（新アプリにもこのファイルをコピーして使い回せる）。要約:

1. `supabase db push`（profiles migration を本番 DB へ）
   — **これより先にデプロイすると全ユーザーが /pending に固定される**
2. Supabase Dashboard → Authentication: Enable Sign Ups **ON** / Confirm email **ON** /
   Site URL=本番ドメイン / Redirect URLs に `https://<本番ドメイン>/auth/confirm`
3. Vercel デプロイ（env は URL と anon key のみ）
4. 初代 admin 昇格（SQL Editor）:
   `update public.profiles set role='admin', status='approved' where email='<管理者メール>';`
5. E2E スモーク（申請→確認メール→pending遮断→承認→利用開始→無効化→再有効化→PWリセット→375px）
   — チェックリストは `認証本番化の適用手順.md` ⑤の A〜J をそのまま使う

ロールバック: Dashboard で Enable Sign Ups を OFF（申請受付の即時停止・既存ユーザー影響なし）。

## 落とし穴チェックリスト（このリポジトリで実際に踏んだ/潰したもの）

- [ ] **適用順**: migration 未適用の本番へ承認ゲートのコードをデプロイすると全員 /pending 化（middleware の fail-safe が働くため）。必ず DB が先
- [ ] **config.toml は本番に効かない**: 本番の Auth 設定は Dashboard 管理。Confirm email OFF のままだと「確認メールを送信しました」の案内と実挙動が乖離する
- [ ] **トリガーの null email**: `coalesce(new.email,'')` を外すと、email が null の signup（将来のソーシャルログイン等）が NOT NULL 違反で全滅する
- [ ] **RLS は列単位制御不可**: admin update を status 系に絞るのはアプリコードの責務。update payload を作る関数（buildStatusUpdate）以外から profiles を update しない
- [ ] **anon の profiles select はエラーになる**（is_admin の execute 権限がないため 0行でなくエラー）。仕様として許容（認証後にのみ照会する）。anon から profiles を読むコードを書かない
- [ ] **Supabase 標準メールのレート制限**（時間あたり数通）。利用者が増えたら SMTP（Resend 等）へ移行（手順概要は `認証本番化の適用手順.md`）
- [ ] **Supabase 障害時は承認済みユーザーも /pending に倒れる**（fail-safe のトレードオフ。少人数運用では許容と判断済み。大規模化するなら JWT claim 化を検討）
- [ ] **`/auth/*` は全状態素通し**が正しい（pending 中のメール確認・recovery リンクを壊さないため）。塞がないこと
- [ ] **承認ゲートの例外パスは最小に**: pending/disabled が通れるのは `/pending`・`/auth/*`・`/reset-password` の3つだけ。増やすときは「未承認ユーザーに見せてよいか」を必ず確認

## 設計判断の背景（なぜこうしたか）

`project-os/knowledge/decisions.md`（2026-06-11「SPEC-0228 / ログイン本番化イニシアチブの認証設計」）参照。要点:
- service role 不使用 = 鍵の漏えい面を増やさず、既存の RLS 中心防御モデルに統一するため
- middleware での DB 照会（JWT claim 不採用）= 承認直後に再ログイン不要な UX と実装単純さを優先（少人数前提）
- 拒否 = `status='disabled'`（auth.users 削除はスコープ外。必要時は Dashboard で実施）
