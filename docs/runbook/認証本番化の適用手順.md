# 認証本番化の適用手順（TKT-0228〜0232 本番適用ゲート）

> この手順書は、TKT-0228〜0232 で実装した承認制ログインを、
> 本番（hosted Supabase + Vercel）で有効にするためのチェックリスト付き運用手順です。
>
> - Supabase 全般の操作基礎・用語は `docs/runbook/Supabaseの反映と運用ガイド.md` を参照してください。
> - 初代 admin の昇格 SQL は同ガイド §10b「承認制ログインと初代 admin の作り方（TKT-0228）」に記載しています。
>   このファイルでは概要と参照先のみを示し、**重複記載はしません**。
> - 秘密情報（APIキー・DB パスワード・個人メールアドレス・本番 URL の実値）はこのファイルに書きません。
>   実値は `.env.local` または Vercel 環境変数でのみ管理してください。

## 0. 適用前に確認すること

この手順は **①→②→③→④→⑤ の順**で上から実施します。
順序を入れ替えると以下のような事故が起きます。

| 誤った順序 | 起きること |
|---|---|
| migration 未適用のまま Vercel にデプロイ | `profiles` テーブルが存在しないため middleware がエラー → 全ユーザーが `/pending` に落ちる |
| Dashboard Auth 設定前に signup ページを公開 | 確認メールが飛ばない・リダイレクト先が 404 になる |
| 初代 admin 昇格前に承認フローを公開 | 誰も申請を承認できない |

## 適用順チェックリスト

### ① migration の確認と適用

> ここだけ先に読んでください。

> **重大な警告**: `20260611120000_auth_profiles.sql` が本番 DB に未適用のまま
> Vercel にコードをデプロイすると、middleware が `profiles` テーブルを参照しようとして
> エラーになり、**承認済みの既存ユーザーを含む全員が `/pending` に固定される**。
> 必ず migration → Vercel デプロイの順で実施すること。

#### 1-1. 未適用 migration の確認

本番（または hosted）プロジェクトに link して現在の適用状況を確認します。

```bash
supabase link --project-ref <本番プロジェクトの参照ID>
supabase migration list
```

`Applied` 列が空になっている（未適用）ファイルが対象です。
以下の **3 つの migration** がすべて本番 DB に適用済みであることを確認してください。

| ファイル名 | 内容 | 関連チケット |
|---|---|---|
| `20260602120000_ai_usage_events.sql` | AI 使用量ログテーブル | TKT-0151 |
| `20260609120000_shopping_items_meal_schedule_link.sql` | 買い物項目と献立スケジュールの連携列 | TKT-0212 |
| `20260611120000_auth_profiles.sql` | profiles テーブル・RLS・トリガー（承認制ログインの土台） | TKT-0228 |

> `shopping_items_meal_schedule_link` が未適用の場合、スケジュール起点の買い物項目追加が失敗します。
> `auth_profiles` が未適用の場合は上記の重大な警告の通りです。

#### 1-2. 内容の事前確認（dry run）

```bash
supabase db push --dry-run
```

流れる予定の SQL が表示されます。意図しない DROP / ALTER / DELETE が含まれていないかを目視確認してください。

#### 1-3. 本番 DB へ適用

```bash
supabase db push
```

完了後、Supabase Studio の **Table Editor** で以下を確認します。

- [ ] `ai_usage_events` テーブルが存在する
- [ ] `shopping_items` テーブルに `meal_schedule_id` 列が存在する
- [ ] `profiles` テーブルが存在し、`status` / `role` / `email` などの列がある
- [ ] 既存ユーザーの行が `status='approved'` で backfill されている（件数が auth.users と一致）
- [ ] 既存データ（在庫・レシピ等）が消えていない

### ② Supabase Dashboard の Auth 設定

Supabase の管理画面（Dashboard）で以下を設定します。
**config.toml はローカル練習場の設定**であり、本番 Dashboard の設定とは別管理です。
Dashboard で変更しないと本番では有効になりません。

画面の場所: Dashboard → （該当プロジェクト）→ **Authentication** → **Settings**

#### 2-1. ユーザー登録の許可

| 設定項目 | 設定値 | 備考 |
|---|---|---|
| Enable Sign Ups | **ON**（有効） | OFF のままだと `/signup` から登録できない |

#### 2-2. メール確認の必須化

| 設定項目 | 設定値 | 備考 |
|---|---|---|
| Confirm email | **ON**（有効） | OFF のままだと確認メールが飛ばず、`/auth/confirm` の案内と乖離する |

> Confirm email が OFF の場合、メール確認待ちと案内しながら実際は確認なしで登録が完了し、
> ユーザー体験が乖離します。

#### 2-3. Site URL とリダイレクト URL

| 設定項目 | 設定値 |
|---|---|
| Site URL | `https://<本番ドメイン>` |
| Redirect URLs（追加） | `https://<本番ドメイン>/auth/confirm` |

> `<本番ドメイン>` には Vercel で発行された本番ドメイン（または独自ドメイン）を入れてください。
> Redirect URLs への追加を忘れると、メール確認リンクをクリックしたときに Supabase が
> 「許可されていないリダイレクト先」としてエラーを返します。

#### 2-4. パスワード最低長の確認

| 設定項目 | 確認値 |
|---|---|
| Minimum password length | TKT-0149 で定めた基準値以上であること |

> 基準値は `docs/runbook/公開前セキュリティ確認.md` を参照してください。

### ③ Vercel へのデプロイ（コード反映）

migration が本番 DB に適用されたことを確認してから Vercel にデプロイします。

- Vercel の環境変数（`NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY`）が
  本番プロジェクトの値になっていることを確認してください。
- `SUPABASE_SERVICE_ROLE_KEY` は **web/ のコードには使用していません**（SPEC-0228 の設計決定）。
  Vercel 環境変数に追加不要です。万一 `NEXT_PUBLIC_` を付けて設定すると世界中に漏洩するため注意してください。
- デプロイ後、本番 URL にアクセスして既存ユーザーで正常ログインできることを確認します。

### ④ 初代 admin への昇格

migration を適用すると既存ユーザーはすべて `status='approved'` / `role='member'` になります。
管理者を1人だけ昇格させるため、Supabase Studio の **SQL Editor** で手作業の SQL を実行します。

詳細な手順とSQL文は
**`docs/runbook/Supabaseの反映と運用ガイド.md` §10b「承認制ログインと初代 admin の作り方（TKT-0228）」**
を参照してください。

概要:
1. `supabase db push` で `20260611120000_auth_profiles.sql` が適用済みであることを確認する（① で実施済み）。
2. Studio の SQL Editor を開く。
3. §10b に記載の UPDATE SQL の `<管理者のメールアドレス>` を実際の値に置き換えて実行する（実値はこのファイルに書かない）。
4. 1 行更新されたことを確認する（0 行の場合は、そのメールのユーザーがまだ存在しないか、綴り違い）。

> admin を複数人設定する場合は同じ SQL を対象メールごとに繰り返します。

### ⑤ E2E手動スモークチェックリスト

migration・Dashboard 設定・デプロイ・admin 昇格が完了したら、以下を1本のシナリオとして上から順に実施してください。
TKT-0228〜0232 の manual-smokes で `skipped_checks`（実機未消化）として持ち越した項目を統合しています。

#### 準備

- admin 用の実アカウント（昇格済み）
- 別ブラウザまたはシークレットウィンドウで操作する「申請者」用の新規メールアドレス（未登録）
- さらに別の「停止対象」用アカウント（または同一申請者で手順内で使い分け）

#### シナリオ（上から順に実施）

**A. 新規申請とメール確認**

- [ ] `/signup` を開き、未登録のメールアドレス＋パスワード（確認入力）で登録申請する
- [ ] 「確認メールを送信しました」の案内が表示される（実際に送信されていること）
- [ ] 受信したメールのリンクをクリックすると `/auth/confirm` を経由して遷移する
- [ ] メール確認直後は `/pending`（承認待ち）画面に固定される

**B. pending ユーザーの遮断確認**

- [ ] `/pending` の状態で在庫・レシピ等のデータ画面の URL を直接入力しても `/pending` に戻される
- [ ] `/pending` の状態で `/api/ai/recipes` や `/api/ai/scan-ingredients` を呼ぼうとしても 403 エラーが返る（未承認のまま AI quota を消費しない）
- [ ] pending ユーザーの `/pending` 画面の文言が「承認待ち」であること

**C. admin による承認**

- [ ] admin アカウントでログインし、`/admin` に入れる
- [ ] 申請一覧に上記 pending ユーザーが表示されている
- [ ] 「承認」ボタンを押すと確認なしで即時承認（または確認パネル経由で承認）される
- [ ] 承認後、申請者のアカウントで再アクセスすると `/`（アプリトップ）に入れる

**D. 利用開始後の通常動作確認**

- [ ] 承認済みユーザーで在庫・レシピ等の画面が正常に表示される
- [ ] backfill された既存ユーザーも従来どおりログインして全画面を使える

**E. 無効化と停止文言**

- [ ] admin が `/admin` から対象ユーザーを「拒否/無効化」する
- [ ] 対象ユーザーが次のアクセスで `/pending`（停止文言「利用停止中」）に固定される
- [ ] 停止状態でデータ画面の URL を直叩きしても `/pending` に戻される

**F. 再有効化**

- [ ] admin が `/admin` で同じユーザーを再度「承認」する
- [ ] 対象ユーザーが再アクセスで `/` に入れる

**G. 管理者権限の境界確認**

- [ ] member（一般）ユーザーで `/admin` に直アクセスすると `/` へ redirect される
- [ ] member セッションの DevTools（ブラウザ開発者ツール）から `profiles` の全行 select → 自分の行のみ返る
- [ ] member セッションから他人の `profiles` 行を update しようとすると失敗（0 行 or エラー）
- [ ] admin ユーザー自身のアカウント行には操作ボタンが表示されない（自己操作抑止）
- [ ] サイドバー・設定の管理画面リンクが member には表示されない

**H. パスワードリセット**

- [ ] 承認済みユーザーで `/forgot-password` を開く
- [ ] 登録済みのメールアドレスを入力して送信すると、成功と同一の画面表示になる
- [ ] 未登録のメールアドレスを入力しても、**登録済みの場合と完全に同じ文言・画面**が表示される（メール存在非露出）
- [ ] 受信したリセットメールのリンクをクリックし `/reset-password` へ遷移する
- [ ] 新しいパスワードを入力して変更すると成功メッセージが表示される
- [ ] 旧パスワードではログインできず、新パスワードでログインできる
- [ ] recovery リンクを再利用・期限切れ後に叩くと `/login` へ遷移する

**I. ログアウト**

- [ ] ログアウトボタンを押すと `/login` へ遷移し、ブラウザバックでデータ画面に入れない

**J. スマートフォン表示（幅 375px）**

- [ ] `/signup` / `/pending` / `/admin` / `/forgot-password` / `/reset-password` の各画面が 375px で表示崩れなし

## ロールバック方針

### 申請受付の即時停止

新規申請を受け付けたくない場合（不正利用・スパム等）は、
Dashboard の **Authentication → Settings → Enable Sign Ups を OFF** にします。

- 既存ユーザーのログインには影響しません。
- OFF にすると `/signup` からの新規登録が Supabase 側でエラーとなります（コードの変更不要）。

### 個別ユーザーの対処

特定ユーザーだけ止める場合は、`/admin` 管理画面から「拒否/無効化」します。
緊急時（管理画面が使えない場合）は、`docs/runbook/Supabaseの反映と運用ガイド.md` §10b に記載の
緊急用 SQL（`set status = 'disabled'`）を Studio の SQL Editor で実行します。

### コードのロールバック

Vercel のデプロイ履歴から前のデプロイに切り戻すことができます。
ただし、適用済みの migration を「なかったこと」にするのは困難です。
migration のロールバックが必要な場合は、打ち消す新しい migration を作成してください（既存ファイルは書き換えない）。

## SMTP 移行の概要（独自メール送信への切り替え）

Supabase の標準メール（Supabase SMTP）は送信数に制限があります。
利用者が増えた場合や独自ドメインからの送信が必要な場合は、外部 SMTP（例: Resend）への移行を検討してください。

移行の大まかな流れ:
1. Resend 等のサービスでアカウントを作成し、送信ドメインを確認する（DNS 設定）。
2. そのサービスの SMTP 認証情報（ホスト・ポート・ユーザー名・パスワード）を取得する。
3. Supabase Dashboard → **Authentication → Settings → SMTP Settings** で
   「Enable Custom SMTP」を ON にし、取得した認証情報を設定する。
4. テスト送信でメールが届くことを確認する。

> 実際の設定値・DNS レコード・サービス固有の手順は、各サービスの公式ドキュメントを参照してください。
> このファイルには実値を記載しません。

Resend 公式: https://resend.com/docs
Supabase 独自 SMTP 設定公式: https://supabase.com/docs/guides/auth/auth-smtp

| 項目 | 内容 | 対処方針 |
