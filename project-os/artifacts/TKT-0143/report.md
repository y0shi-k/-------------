# Report Draft

## 変更目的

Web版「献立・レシピ > スケジュール」画面の見た目がCanvas版（正）と大きく違っていたため、レイアウト・配色・余白・ボタン形状・カード構造・追加導線・週ナビをCanvas版に一致させた。あわせて、予定が入ったカードの操作の見せ方と、買い物まわりの置き場所もCanvas版に合わせた。データの保存処理（Supabase）はそのまま維持している。

## 今回追加した安全装置

- 献立の追加処理（Supabaseへの保存）は従来と同じテーブル・項目・本人ID付きで行う関数に切り出し、見た目を変えても保存の中身が変わらないようにした。
- 予定が入ったカードの操作（前日/翌日/調理完了/削除）は、狭いセル内に展開するとはみ出してレイアウトが崩れていたため、Canvas版同様にカードをタップして開く中央モーダル（献立の操作）へ移した。各操作の中身と消費量調整は従来の動きを引き継いだ。
- Canvas版に合わせ、買い物まわり（不足食材・買い物リスト）をスケジュール画面から外し、買い物リストの管理（手動追加・購入済・選択削除・一覧）を「食材管理」タブへ移設した。レシピ詳細からの不足材料→買い物追加は従来どおり残した。
- 買い物の保存・購入済・削除はいずれも本人ID付きで従来と同じテーブルに対して行う。画面間の表示同期のため書き込み後に再読込（router.refresh）を呼ぶ。
- 追加・移動・削除・調理完了・買い物の手動追加/購入済/一括削除が正しく動くことを自動テストで確認できる状態にした（テスト52件すべて成功）。
- データベースの構造・権限（RLS）・写真保管（Storage）・ログイン（Auth）は一切変更していないことを、別紙のmanual-smokes / reviewで明示確認した。
- Canvas版 `app.html` は参照のみで変更していない。

## 実施した確認

- `npm run lint` — エラー・警告なし
- `npm run typecheck` — 型エラーなし
- `npm run test` — 11ファイル / 52テスト すべて成功（スケジュールの追加・カード操作モーダル経由の移動/削除/調理完了、食材管理タブでの買い物の手動追加/購入済/一括削除を検証）
- `npm run build` — ビルド成功
- policyチェック — GAS非依存 / 秘密情報の直書きなし / RLS確認 すべてpass（verify.json）

## 残リスク

- スマホ実機（ログイン済み・本番相当データ）での見た目とカード操作モーダルの最終目視は未実施。
- 買い物の表示は画面ごとに自前の状態を持つ（食材管理タブとレシピ詳細）。書き込み後はrouter.refreshで再読込して同期するが、同一セッションで両画面を行き来する際の即時反映はモード切替時の再マウントに依存する。
- ドラッグでの並べ替えと「選択モードでの一括削除」は今回の範囲外（移動は献立の操作モーダルの前日/翌日で可能）。

## 次の依頼や人判断

- 「選択モードでの一括削除」をCanvas版同様に追加するかは、別チケットで判断したい。
- 食材管理タブの買い物リストにあった「出自別／材料まとめ」切替は、機能実装まで未対応（今回は手動追加・購入済・選択削除・一覧の基本操作を実装）。必要なら別チケット化。
- 実機（ログイン済み）での見た目最終確認は、本番相当データで一度目視してほしい。
