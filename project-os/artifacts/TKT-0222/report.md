# Report Draft

> この artifact は日本語で記述する。コマンド名・ファイルパス・status値などの識別子だけ英語を許容する。

## 変更目的

「たまご」と「卵」のような表記ゆれでレシピ食材と在庫が紐付かない問題（SPEC-0222）の土台として、共通の名前マッチングユーティリティを新設した。UIへの適用は TKT-0223（消費量調整）/ TKT-0224（買い物不足計算）で行う。

- `web/src/lib/ingredients/name-match.ts`（新規）:
  - `normalizeIngredientMatchName(name)`: NFKC・小文字化・空白除去・カタカナ→ひらがな（U+30A1〜U+30F6 を -0x60。長音「ー」は保持）。
  - `matchesIngredientName(a, b)`: 正規化一致 or 類義語辞書一致のみ true。**部分一致では true を返さない**（買い忘れ防止のユーザー確定方針）。
  - `ingredientNameMatchScore(a, b)`: 4=完全一致 > 3=正規化一致 > 2=辞書一致 > 1=部分一致 > 0=不一致。matches は 2 以上のみ。
  - 類義語辞書は20グループ（卵/玉ねぎ/じゃがいも/さつまいも/ねぎ/にんじん/トマト/大根/きゅうり/ほうれん草/キャベツ/豆腐/鶏肉/牛乳/砂糖/醤油/味噌/バター/小麦粉/もやし）。モジュール初期化時に「正規化済み語→グループID」Map を構築し O(1) 照合。ピーマン/パプリカは同一視しない。
- `web/src/__tests__/ingredient-name-match.test.ts`（新規）: 45テスト。

## 今回追加した安全装置

- `matchesIngredientName` が部分一致だけでは true を返さないことをテストで固定（SPEC-0222 の核となる境界）。
- 部分一致（score=1）は「正規化後の包含」に加え「短い方が長い方の部分列（2文字以上）」も対象（例: 豚肉↔豚こま切れ肉）。**レビューでの修正点**: 実装サブエージェントは包含のみで実装し acceptance の「豚肉/豚こま切れ肉 score>不一致」をテスト側を変えて回避していたため、オーケストレーターが部分列判定を追加して acceptance どおりに戻した。部分列は候補の並び順にのみ使われ、同一視・不足計算には使われないため過剰マッチの実害はない。
- 全関数は純粋関数・引数非破壊。既存の画像用 `normalizeIngredientName`（`ingredient-image.ts`）には触れていない。

## 残リスク

- 辞書は静的20グループで網羅的ではない。運用で語彙を足す前提（ユーザー編集UIは非ゴール、将来検討）。
