# TKT-0148 実装報告

## 結論

献立からの調理完了フローで、消費量調整モーダルを調理ビューアより前面に表示し、チェック選択・全選択/全解除・カテゴリタブ・在庫不足警告・完成写真・評価・コメントを同じモーダルに統合した。

## 変更内容

- `web/src/components/recipe-meal-workspace.tsx`
  - 消費量ドラフトに `selected` を追加し、チェック済みの行だけ在庫減算と消費イベント保存の対象にした。
  - 消費量モーダルに「全 / 食材 / 調味料」タブ、全選択/全解除、在庫不足警告を追加した。
  - 完成写真、評価★1〜5、一言コメント入力を追加した。
  - 料理履歴へ `rating` と `note` を保存するようにした。
  - 写真がある場合は圧縮後に Supabase Storage `photos` バケットへ保存し、`photos` テーブルに `cooking_history_id` 付きで紐付けるようにした。
  - 写真保存だけ失敗した場合は、在庫減算・献立完了・料理履歴作成を保持し、警告トーストを出すようにした。
- `web/src/app/globals.css`
  - `.consumption-backdrop` の `z-index` を `90` にして、`.cooking-overlay` の `85` より前面に出した。
  - 追加した消費量UI、写真、評価、コメントのスタイルを追加した。
- `web/src/__tests__/recipe-meal-workspace.test.tsx`
  - チェック解除時に在庫減算しないこと、評価とコメントが料理履歴へ保存されることを確認するテストを追加した。

## セキュリティ

- APIキーや秘密鍵の直書きはしていない。
- 新しいバケットや公開URLは追加していない。
- 写真パスは既存の `buildCookingHistoryPhotoStoragePath(userId)` を使い、ユーザーID配下に保存する。
- GAS / Spreadsheet / Drive 依存は追加していない。

## 残リスク

- 在庫減算は既存どおり複数行の逐次更新で、途中失敗時に部分更新の可能性がある。このチケットでは既存仕様を維持した。
- 単位変換は対象外。同じ単位の在庫だけを代替候補に出す。
- ログイン後の実データを使った写真アップロード確認は、Supabase Storage 接続と端末カメラが必要。
