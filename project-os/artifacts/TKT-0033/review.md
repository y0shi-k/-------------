# TKT-0033 セルフレビュー

## 良い点

- スプレッドシートの既存データ移行をGAS側で自動化し、ユーザー操作を必要としない設計
- IDベースの操作に移行しつつ、旧データ（idなし）へのフォールバックを保持
- D&D後の個別GAS通信を行わず、`pendingSync` + `updateSyncBar()` のパターンに従った
- `alert`/`confirm`/`prompt` を使用せず、独自モーダルで確認ダイアログを実現
- `createClientId('sched')` を使用し、既存のID生成パターンに従った

## 懸念事項・改善余地

### 1. D&Dの同ブロック並び替え精度

マウスY座標からカード挿入位置を推定しているが、カード高さを固定値（52px）で計算している。フォントサイズやデバイスによってずれる可能性がある。実際の使用感で問題が出た場合、ドラッグオーバー時の挿入インジケータ（水平線）を表示する方式に改善すべき。

### 2. sortOrderの値域

同ブロック内D&Dで `index * 10` を設定している。頻繁にD&Dを繰り返すと、新しい値が大きくなりすぎる可能性がある（理論上はNumber.MAX_SAFE_INTEGERまで問題ない）。将来的に「整理」ボタンで0からの連番に正規化する機能を追加しても良い。

### 3. レシピ変更時の動作

`changeScheduleSlotRecipe`（別レシピに変更）では、旧アイテムをdeleteして新アイテムをcreateする方式にした。これはIDが変わるため、同じ献立スロットを「履歴として追跡」したい場合には不向き。ただし現状の仕様では「別レシピに変更」は別レシピとして扱うのが自然なので、設計として妥当。

### 4. 選択モード時のD&D

選択モードON時もD&Dは有効になっている。ドラッグハンドル（≡）でのみ開始するので、チェックボックス操作と競合しないが、ユーザーの意図しないD&Dが発生する可能性がある。問題が出た場合、選択モード時は `draggable=false` にする改善を検討。

### 5. GAS側の旧データフォールバック

`scheduleDeletes` / `scheduleUpdates` で `!del.id` の場合に `date+meal+recipeId` でマッチングしているが、旧データの `recipeId` が空の場合（あり得ないはずだが）に誤削除のリスクがある。実運用上は問題ないと判断。

## 結論

主要な懸念はD&Dの精度とsortOrderの値域だが、いずれも実用上問題ないレベル。必要に応じて後続チケットで改善可能。
