## 2026-06-04（ホーム/ダッシュボード新設 / TKT-0162）

### 決定
PCデスクトップUI化（案A）の最終ピースとして、PC向けホーム/ダッシュボードを新設し完了した。
- ホームの中身は **page.tsx で組み立てて `home` ReactNode として WebModeShell に渡す**（childrenByMode と同じ設計）。shell はデータ非依存に保ち、描画スロットと状態遷移のみ担う。
- 中身は **既存データの集約のみ**: 挨拶＋サマリーカード4枚（既存カウント）＋既存 `TodayDashboard`（今日の確認）。新規DBクエリ・新規データソースは追加しない。
- **PC初期表示=ホーム**は、初期 state を ingredients に据え置き、マウント後 `window.matchMedia("(min-width: 1024px)")` が真のときだけ home へ昇格する方式にした。スマホは昇格せず食材管理起点・下部タブ3つ据え置き。

### 理由
- SSR/ハイドレーション不整合を避けつつ、PCのみ初期ホーム・スマホは従来動線、という案A要件を両立できる。jsdom（matchMedia 未定義）では昇格しないため既存テスト（ingredients 初期前提）も無改変で通る。
- shell をデータ非依存にすることで、ホームの集約ロジックを page.tsx 側に閉じ、shell の責務肥大を防ぐ。

### 却下した案
- **初期 state を home に固定** → SSR で mobile も home になり「スマホ食材管理起点」を満たせない。ハイドレーション不整合・既存テスト破壊のリスクもあり却下。
- **TodayDashboard を作り直す/独自ウィジェット追加** → 既存 `TodayDashboard` が今日の献立・在庫アラート・買い物・候補を既に持つため再利用。飾りウィジェットは作らない方針。

### 次に確認すること
- ブラウザ目視（PC初回ホーム・カード遷移、スマホ食材管理起点・ホーム非表示）。
- `supabase_schema_change` 🔴 はテーブル名トークンの過剰マッチ（実 schema/RLS/Storage 無変更）。manual-smokes.md / review.md に静的確認を記録済み（TKT-0160/0166 と同様）。

## 2026-06-04（ビジュアルレイヤー導入 / TKT-0168〜0172）

### 決定
ホームが参考モック（`レシピイメージ図`）と程遠い件への対応として、**写真・絵文字のビジュアルレイヤー**を全画面に導入する。方針は **レイアウト先行＋6画面まとめて**（ユーザー確定）。
- **レシピ写真は当面 `web/public/` の静的デモ画像**を名前→画像で解決（`recipe-image.ts`）。**schema / Supabase Storage は変更しない**。将来 `recipes.image_path` へ差し替えてもI/F不変の設計にする。
- **食材は絵文字**（`ingredient-emoji.ts`、画像生成不要）。
- 画像は **Codex(DALL·E) 生成**。発注書 `docs/design/demo-image-assets.md`、設計は正本 §8（`docs/design/pc-design-language.md`）。
- 統一感の担保として、各画面は独自に `<img>`/絵文字を直書きせず、**基盤チケット TKT-0168 の共通 resolver/コンポーネント（`<RecipeThumb>`/`<IngredientIcon>`）を必ず経由**する。TKT-0168 を 0169〜0172 の前提（依存元）にする。
- 全チケット非危険変更（CSS/JSX/静的アセット）。設定画面は画像不要のため対象外。

### 理由
- 参考図との根本差は「画像の有無＋写真を主役にする構造」。写真だけ足しても収まらないため、共通土台→各画面の段階適用にした。
- 静的画像＋絵文字に寄せることで、schema/Storage/auth の危険変更を避けつつ見た目を大きく前進させられる（プロトタイプとして妥当）。
- 別セッションでの実装でもトーンが揃うよう、共通部品を1箇所（TKT-0168）に集約した。

### 却下した案
- **画像先行**（Codex画像を待ってから実装）→ 着手が画像生成に律速されるため見送り。フォールバック付きでレイアウト先行を選択。
- **`recipes.image_path` schema＋Storage を今入れる** → 危険変更（schema/Storage）で重い。プロトタイプ段階では静的mapで十分なため将来送り。
- 食材を画像生成 → 絵文字で十分かつ無コスト。生成は料理写真＋ヒーローに限定。

### 次に確認すること
- TKT-0168 実装後、フォールバック（画像ゼロ）で全画面が崩れないこと。
- `photo_upload_storage` / `ai_server_route` / `csv_import_migration` eval の過剰マッチ（語彙由来・実Storage/移行無し）を各 report に記録。
- TKT-0172 で実画像配置後の見栄え・画像合計サイズ。

## 2026-06-04（TKT-0161 設定画面）

### 決定
散在設定をPCサイドバー下部ギア／スマホはステータスバーのアカウント表示タップから開く設定画面に集約（案A・主ナビ非昇格を実装）。
- APIキー入力は設定画面を唯一の入力場所に**移設**し、各ボードのインラインパネルは撤去。ボードはマウント時 `loadUserGeminiApiKey()` で localStorage から読み込み（旧 `GeminiApiKeyPanel` の useEffect 依存を代替）。AI実行のキー必須ガードは温存。
- AI残量メーターは上部バー/ステータスバー＋設定画面に**集約**し、各ボードパネル内の重複メーターは撤去。
- ログアウトは設定画面に「アカウント」セクションを新設しつつ、PC上部バーの `desktop-account`（メール＋ログアウト）は**据え置き＝併存**とする。

### 理由
ログアウトは元々重複しておらず、PC上部バーはクイックアクセスとして有用。設定への一本化と、認証回帰リスク回避・既存テスト温存を両立させるため併存を選択。APIキー/残量は実際に重複していたため移設・集約で「設定の所在」を一本化。

### 却下した案
- ログアウトをPC上部バーから撤去し設定のみに一本化 → 認証導線の回帰リスクと既存テスト破壊を避けるため見送り（クイックアクセスの利便も優先）。
- APIキー入力をボードにも併存 → 「設定の所在の一本化」という目的に反するため移設を選択。

### 次に確認すること
- 実機ブラウザでの設定遷移・APIキー保存→AI実行・ログアウト・スマホ全画面化の目視（manual-smokes の skipped_checks）。
- `.photo-empty` 文言（「入力したGemini APIキーで」）を設定移設後の表現へ更新するか。

## 2026-06-04

### 決定
レシピのお気に入りを `recipes.is_favorite`（boolean not null default false）で新設（TKT-0167）。
新規 RLS ポリシーは追加せず、既存行ポリシー（`auth.uid() = user_id`）で参照・更新を自分の行に限定する。
切替は専用更新（楽観的更新＋失敗ロールバック）にし、既存の `saveRecipe` payload には含めない。
カテゴリ絞り込みは今回「お気に入り」チップのみ追加（「よく作る」は cook_count 由来で導出可だが今回スコープ外）。
ローカルに Docker が無く真のローカル Supabase が使えないため、ユーザー明示承認のうえ migration を
hosted プロジェクト（`wwtompvneobysieofxkl`）へ `supabase db push` で適用し、schema/RLS/後方互換を
hosted で実検証した。

### 理由
列追加＋default false は加算的で後方互換。RLS は行ポリシーが列にも効くため新規不要。トグルを保存処理から
分離することで既存保存の回帰を避けられる。ローカル隔離環境が無い以上、危険変更の本体（schema/RLS）は
実 DB で確認するのが最も確実で、追加列は低リスク。

### 却下した案
- お気に入り用に新規 RLS ポリシーを追加 → 既存行ポリシーで充足のため不要。
- is_favorite を `saveRecipe` の payload に混ぜる → 保存処理の回帰リスク。トグル専用更新に分離。
- 「よく作る」チップも同時追加 → スコープを最小に保つため見送り（cook_count 由来でいつでも追加可）。
- DB 適用を保留しコードのみ確定 → ローカル DB が無く UI happy-path も確認できないため、ユーザー承認のうえ hosted 適用を選択。

### 付随修正
- Stop hook `hook_decisions_reminder.py` が Stop 非対応の `hookSpecificOutput.additionalContext` を出して
  スキーマエラーになっていたため、`systemMessage` に修正。

### 次に確認すること
- UI happy-path（ハート保存・リロード保持・絞り込み・オフライン時ロールバック・スマホ・ハートの見た目）の
  ブラウザ確認（ユーザー）。崩れ・サイズの微調整要否。

### 決定（PC幅の献立スケジュールは縦アジェンダ表示: TKT-0165）
PC幅（>=1024px）の献立スケジュールは、TKT-0159 で入れた「7列ウィークグリッド」をやめ、スマホ版と同じ
**「1日=1行」の縦アジェンダ表示**（日付バッジ＋朝・昼・晩の横並びスロット）に統一する。盤面は
`width: min(760px, 100%); margin: 0 auto;` で中央寄せ・最大760pxに制限する。

### 理由
7列グリッドは各日カードが `align-items: stretch` + `flex:1` で盤面全高まで引き伸ばされ、1日最大3枠しか
ないデータに対しスロット下に大きな縦余白（死に領域）が出た（TKT-0157 の `desktop-workspace` 全高伸長と
噛み合った副作用）。横に広げるメリットよりも余白の無駄が大きく、縦アジェンダの方が見た目・可読性とも良い。

- 7列を維持しつつ高さを内容に合わせる（stretch解除・min-height調整）→ それでも横幅が間延びし、1日3枠に対し
  カラムが空きすぎる。根本的に7列が過剰と判断。

  触らず最小差分で済む方を採用。

### 次に確認すること

### 決定（完了モーダルの除外を「消費量0」に一本化: TKT-0164）
「料理を完了する」完了モーダル（`ConsumptionEditor`）で、在庫減算の除外手段を **「消費量0」だけ** に一本化する。
チェックボックス（`selected`）と在庫セレクトの「減算しない」option を廃止し、材料名＋在庫select＋消費量(+単位)を
横一列にする高密度レイアウトへ刷新（消費量入力は `NumberField`）。一括操作は「全部 既定量／全部 0」。

### 理由
完了モーダルは一度きりの減算で、保存ゲート `!draft.selected || !draft.stockItemId || draft.consumedAmount === 0`
の3条件が同義の除外手段として冗長に重なり、UIを縦長・複雑にしていた（ユーザー指摘: 「消費量0なら減らさないのでは」）。
「消費量0で減算しない」は保存ロジック上も保証されているため、最もシンプルな1軸に統一できる。

### 却下した案
- チェックボックスを残す案 → 「全選択/全解除」のための快速トグルとして有用だが、除外軸を2つ残し冗長。ユーザーが最小化を選択。
- 料理履歴編集モーダル（`cooking-record-edit-modal.tsx`）も同時に統一する案 → そこではチェックOFF＝**消費明細のDB削除＋在庫差分の戻し**、
  「減算しない」＝**在庫非紐づけだが記録は残す**という固有意味があり、統一すると在庫差分計算を壊す**危険変更**。本チケットでは無改変とし、別チケット候補に。
- 共有CSS `.consumption-item` を改修する案 → 編集モーダルと結合するため却下。完了モーダル専用の新規クラス `.consumption-row` を追加。

### 次に確認すること
- 実機での高密度レイアウトの見やすさ／狭幅での折返し／全角→半角の挙動（manual-smokes の open_risks）。
- 編集モーダルの見た目統一を別チケットで行うか（危険変更として review/manual-smokes 必須）。

## 2026-06-03

### 決定（PCデスクトップUI化のIA: TKT-0157〜0163）
PC（≥1024px）を添付モック準拠の「左サイドバー＋上部バー＋多カラム」デスクトップシェルにする。
**スマホ（<1024px）は現状を完全温存**（下部タブ3つ・中央660px列・上部ステータスバー）。
7チケットに分割（土台 TKT-0157 → 各モード 0158-0160 → 拡張 0161-0162 → 任意 0163）。

確定したIA（ユーザー確認済み 2026-06-03）:
- **設定（TKT-0161）= 案A**: 設定は主ナビに昇格させず、**PCはサイドバー下部にギア固定**、**スマホはアカウント導線**から全画面の設定へ。散在する Gemini APIキー／AI残量／ログアウトを設定画面に集約。主ナビ3つ・スマホ下部タブ3つを温存。
- **ホーム（TKT-0162）= 案A**: 「ようこそ」ホームを新設し **PCのみ初期表示**（PCサイドバーの主ナビに「ホーム」を追加）。**スマホは従来どおり食材管理起点**で、下部タブは増やさない。ホームは既存データの集約表示のみ（新クエリを作らない）。
- **PCサイドバーのIA = 案A（グループ化ツリー）**: 現状の3モードを**セクション見出し**にし、各モードが内部に持つタブを**葉(leaf)**として展開する2階層ナビにする。
  - 構造: 🏠ホーム / [食材管理: 在庫一覧・買い物リスト] / [献立・レシピ: レシピ・献立スケジュール] / [料理・記録: カレンダー・タイムライン・インサイト] / ⚙設定（下部固定）。
  - **目的地（常駐画面）だけを葉にする**。アクション（＋追加・AI食材登録・レシピ生成・テキスト取り込み）は葉にせずコンテンツ内のボタン/モーダルのまま。「今日の献立」はホーム(0162)に集約しサイドバーには出さない。
  - グループ見出しクリックはそのグループの既定の葉へ（例: 食材管理→在庫一覧）。約8葉のため常時展開・折りたたみなし。
  - **同一論理ツリーの二面レンダリング**: スマホ=「3モード＋内部タブ」、PC=「3グループ＋葉に展開」。IAは分岐させず単一ツリーとして管理する。
  - 実装: シェルが選択中の (group, leaf) を保持し、各ボードはサブビュー状態を**controlled/initial prop**として受け取れるよう持ち上げる（小リファクタ）。PCではコンテンツ内のタブ列（`.recipe-subnav` / `.cooking-view-tabs` / 食材の表示切替）を隠しサイドバーに一本化。スマホは内部タブのまま。→ TKT-0157＋0158/0159/0160 に波及。
- スマホ下部タブは従来どおり3つ（食材管理/献立・レシピ/料理・記録）を温存。
- **献立スケジュール（TKT-0159）= 案A**: PCは「曜日=列/朝・昼・夜=行」の7日ウィークグリッドへ転置（既存の今日±3日ウィンドウ模型・±週ナビ・ドラッグ&ドロップ結線は不変、並べ方のみ変更）。スマホは現状の縦アジェンダを維持。サイドバーでは「献立・レシピ」グループ直下の葉として独立させる。
- **料理記録カレンダー（TKT-0160）**: 既に7列の月グリッドのため構造変更なし。大画面ではセル・写真サムネを拡大して見やすくするのみ（判断不要）。

### 理由
パレット・角丸・影・紫アクセントは既にモックと一致しており、作業の主眼は「見た目の作り替え」ではなく「PC用シェル（骨格）の追加」。スマホを温存して影響範囲を限定するため、デスクトップ表示はブレークポイントで分岐させる。
設定は使用頻度が低く主ナビの一等地を割く価値が薄いためサイドバー下部へ。ホームは在庫アラート・今日の献立・レシピ候補を一目で把握する価値があるためPC既定にするが、スマホは即タスクの軽快さを優先して据え置く。

### 方針の見直し（TKT-0138 を上書き）
TKT-0138 は「PCに広げず Canvas のモバイル型へ寄せる／独自のPC向けアレンジを入れない」方針だった。
本決定は **≥1024px のデスクトップ表示についてのみ**その方針を見直す（モバイル表示は TKT-0138 の意図どおり温存）。

### 却下した案
- 設定を主ナビ第4項目に昇格（案C） → スマホ下部タブが4つで窮屈、主ナビが薄まるため却下。
- ホームを独立させず食材管理上部に挨拶＋サマリーを差し込む（案C） → 最小実装だがモックの独立ダッシュボードと別物・拡張余地が小さい。軽量案として保留。

### 決定（TKT-0151: Gemini AI利用の1日回数制限）
Gemini AI機能に日次上限（recipe 20 / scan 10 / 合計 30、JST暦日）をサーバー側で設ける。
判定は **原子的予約方式**: Gemini送信前に `consume_ai_usage()`（`pg_advisory_xact_lock` + 条件付きinsert）で
1回分を予約し、上限超過時はGeminiへ送らず429。Gemini通信失敗・写真DL失敗時は `refund_ai_usage()` で返金し
当日枠を消費しない。ok応答後のparse失敗は消費維持。`ai_usage_events` はRLS select本人のみ、
insert/update/delete は閉じ、書き込みは SECURITY DEFINER 関数経由（authenticatedのみ grant）。
refundは本人の直近5分の予約のみ削除可（履歴削除によるリセット不可）。上限値はSQL関数に単一定義。

### 理由
TKT-0150でAPIキーがユーザー持ちになり、上限の主目的は「運営者費用の抑制」→「乗っ取り・連打時の
本人課金の暴走防止の安全ネット」に変質した。連打耐性（原子的予約）と「一時的Geminiエラーで枠を
消費しない」（返金）を両立させるため、予約＋失敗時返金を採用。

### 却下した案
- 「成功時のみ記録（予約なし）」 → 大量同時リクエスト（乗っ取りバースト）に弱く、上限を一時的に超えうる。
- 「予約・失敗もカウント」 → 実装は最小だが、一時的502で自費キーの枠を消費しUX悪化。
- 上限値をlib/UIにも定義 → ドリフトの元。SQL単一ソースに統一。
- 日付境界UTC → DB実装は単純だがユーザー体験はJSTが自然。JSTを採用。

### 次に確認すること
- 公開前に `supabase db push` で適用し、実DBで上限到達・返金・別ユーザー分離・スマホ表示を手動確認。
- refundの5分制限が実Gemini最大応答時間と整合するか。

### 決定（SPEC-0228 / ログイン本番化イニシアチブの認証設計）
新規ユーザー受け入れを「申請＋管理者承認」方式で確定（ユーザー確認済み）。コア設計:
- `public.profiles`（status: pending/approved/disabled、role: member/admin）を auth.users トリガーで自動作成。既存ユーザーは backfill で approved。
- **service role key は web/ に持ち込まない**。管理操作（承認/拒否/無効化）は admin RLS ポリシー（`is_admin()` security definer）経由のクライアント直 update。
- 承認状態の判定は middleware で `getUser()` 後に profiles.status を1クエリ参照（4状態: 未ログイン/pending/disabled/approved）。JWT claim 化・キャッシュは少人数運用では不採用。
- 拒否=`status='disabled'`（auth.users の削除はスコープ外。必要時は Dashboard）。
- メールは当面 Supabase 標準（SMTP 移行手順のみ TKT-0233 runbook に残す）。

### 理由
service role の漏えい面を増やさず、既存の RLS 中心設計（全テーブル `auth.uid() = user_id`）と同じ防御モデルで承認制を実現できるため。middleware は既に毎リクエスト `getUser()` を呼んでおり、+1クエリは許容範囲。

### 却下した案
- 招待制のみ → 申請の受け口が欲しいというユーザー要望に不一致。
- service role を使う admin API route → 鍵管理・漏えい面が増える。RLS で十分なため却下。
- JWT app_metadata に status/role を埋める → 承認直後に再ログインが必要になる UX と実装複雑度から、少人数運用では DB 参照を採用。

### 次に確認すること
- TKT-0231（管理画面）の review で「admin update ポリシーの列単位制御不可」を UI ガードで担保できているか確認。
- hosted 適用（TKT-0233 ゲート）で manual-smokes の skipped_checks を実機消化。
