## 2026-06-03

### 決定（PCデザイン言語の正本を新設し Image #3 トーンに統一）
PC幅（>=1024px）の見た目を、モック画像 Image #3（クリーン・余白広め・indigo+白基調）のトーンに統一する。
そのための設計正本として `docs/design/pc-design-language.md` を作成し、今後のPC各画面の実装・レビューの基準にする。
場当たり的なCSS修正の前に、デザイントークン（カラー/タイポ/角丸/影/スペーシング）とコンポーネント規定を先に定義した。

当面の意思決定（ユーザー確認済み）:
- レシピ写真: 当面プレースホルダ（写真なし）。差し込み口だけ用意。schema/Storage変更なし。
- 調理時間: 今回は出さない。schema変更なし。
- お気に入り: `recipes.is_favorite` を新設（schema変更=危険変更。専用チケットで manual-smokes/review 必須）。
- 適用順: ①土台トークン → ②レシピカード → ③お気に入りschema → ④献立配色 → ⑤在庫/料理記録/設定/ホーム。スマホは温存。

### 理由
レシピカード（Image #2）が「名前1文字省略・操作ボタン過密・タグ溢れ」で崩れ、献立も含めPC content 側が
sky-blue多用・多色で Image #3 の静かなトーンと不一致。画面ごとに場当たり修正すると不整合が増えるため、
先にトーンの正本を1枚に固定し、各画面はそれに収束させる方針にした。

### 却下した案
- design.md を作らずレシピカードだけ直す → 他画面との不整合が残り、同じ議論を繰り返すため却下。
- モック完全準拠を1チケットで一括実装 → 写真/お気に入り(schema)/各画面を1本に混ぜると危険変更と非危険が
  混在し巨大化。土台→各画面の段階適用（危険変更は分離）に分割した。

### 次に確認すること
- レシピカード実装時、ジャンルタグの「彩度を落とす」具体値（淡色 or indigo単色化）を実画面で確定。
- お気に入り schema チケットの本番適用（`supabase db push`）タイミング。

### 決定（PC幅の献立スケジュールは縦アジェンダ表示: TKT-0165）
PC幅（>=1024px）の献立スケジュールは、TKT-0159 で入れた「7列ウィークグリッド」をやめ、スマホ版と同じ
**「1日=1行」の縦アジェンダ表示**（日付バッジ＋朝・昼・晩の横並びスロット）に統一する。盤面は
`width: min(760px, 100%); margin: 0 auto;` で中央寄せ・最大760pxに制限する。

### 理由
7列グリッドは各日カードが `align-items: stretch` + `flex:1` で盤面全高まで引き伸ばされ、1日最大3枠しか
ないデータに対しスロット下に大きな縦余白（死に領域）が出た（TKT-0157 の `desktop-workspace` 全高伸長と
噛み合った副作用）。横に広げるメリットよりも余白の無駄が大きく、縦アジェンダの方が見た目・可読性とも良い。

### 却下した案
- 7列を維持しつつ高さを内容に合わせる（stretch解除・min-height調整）→ それでも横幅が間延びし、1日3枠に対し
  カラムが空きすぎる。根本的に7列が過剰と判断。
- JSX側でPC専用の別レイアウトを組む → CSSのフォールバックだけで縦アジェンダに戻せるため、DOM・保存処理を
  触らず最小差分で済む方を採用。

### 次に確認すること
- 実機でPC/スマホの見た目と余白解消、760px最大幅の好み（広狭）を確認する。

### 決定（完了モーダルの除外を「消費量0」に一本化: TKT-0164）
「料理を完了する」完了モーダル（`ConsumptionEditor`）で、在庫減算の除外手段を **「消費量0」だけ** に一本化する。
チェックボックス（`selected`）と在庫セレクトの「減算しない」option を廃止し、材料名＋在庫select＋消費量(+単位)を
横一列にする高密度レイアウトへ刷新（消費量入力は `NumberField`）。一括操作は「全部 既定量／全部 0」。

### 理由
完了モーダルは一度きりの減算で、保存ゲート `!draft.selected || !draft.stockItemId || draft.consumedAmount === 0`
の3条件が同義の除外手段として冗長に重なり、UIを縦長・複雑にしていた（ユーザー指摘: 「消費量0なら減らさないのでは」）。
「消費量0で減算しない」は保存ロジック上も保証されているため、最もシンプルな1軸に統一できる。

### 却下した案
- チェックボックスを残す案 → 「全選択/全解除」のための快速トグルとして有用だが、除外軸を2つ残し冗長。ユーザーが最小化を選択。
- 料理履歴編集モーダル（`cooking-record-edit-modal.tsx`）も同時に統一する案 → そこではチェックOFF＝**消費明細のDB削除＋在庫差分の戻し**、
  「減算しない」＝**在庫非紐づけだが記録は残す**という固有意味があり、統一すると在庫差分計算を壊す**危険変更**。本チケットでは無改変とし、別チケット候補に。
- 共有CSS `.consumption-item` を改修する案 → 編集モーダルと結合するため却下。完了モーダル専用の新規クラス `.consumption-row` を追加。

### 次に確認すること
- 実機での高密度レイアウトの見やすさ／狭幅での折返し／全角→半角の挙動（manual-smokes の open_risks）。
- 編集モーダルの見た目統一を別チケットで行うか（危険変更として review/manual-smokes 必須）。

### 決定（PCデスクトップUI化のIA: TKT-0157〜0163）
PC（≥1024px）を添付モック準拠の「左サイドバー＋上部バー＋多カラム」デスクトップシェルにする。
**スマホ（<1024px）は現状を完全温存**（下部タブ3つ・中央660px列・上部ステータスバー）。
7チケットに分割（土台 TKT-0157 → 各モード 0158-0160 → 拡張 0161-0162 → 任意 0163）。

確定したIA（ユーザー確認済み 2026-06-03）:
- **設定（TKT-0161）= 案A**: 設定は主ナビに昇格させず、**PCはサイドバー下部にギア固定**、**スマホはアカウント導線**から全画面の設定へ。散在する Gemini APIキー／AI残量／ログアウトを設定画面に集約。主ナビ3つ・スマホ下部タブ3つを温存。
- **ホーム（TKT-0162）= 案A**: 「ようこそ」ホームを新設し **PCのみ初期表示**（PCサイドバーの主ナビに「ホーム」を追加）。**スマホは従来どおり食材管理起点**で、下部タブは増やさない。ホームは既存データの集約表示のみ（新クエリを作らない）。
- **PCサイドバーのIA = 案A（グループ化ツリー）**: 現状の3モードを**セクション見出し**にし、各モードが内部に持つタブを**葉(leaf)**として展開する2階層ナビにする。
  - 構造: 🏠ホーム / [食材管理: 在庫一覧・買い物リスト] / [献立・レシピ: レシピ・献立スケジュール] / [料理・記録: カレンダー・タイムライン・インサイト] / ⚙設定（下部固定）。
  - **目的地（常駐画面）だけを葉にする**。アクション（＋追加・AI食材登録・レシピ生成・テキスト取り込み）は葉にせずコンテンツ内のボタン/モーダルのまま。「今日の献立」はホーム(0162)に集約しサイドバーには出さない。
  - グループ見出しクリックはそのグループの既定の葉へ（例: 食材管理→在庫一覧）。約8葉のため常時展開・折りたたみなし。
  - **同一論理ツリーの二面レンダリング**: スマホ=「3モード＋内部タブ」、PC=「3グループ＋葉に展開」。IAは分岐させず単一ツリーとして管理する。
  - 実装: シェルが選択中の (group, leaf) を保持し、各ボードはサブビュー状態を**controlled/initial prop**として受け取れるよう持ち上げる（小リファクタ）。PCではコンテンツ内のタブ列（`.recipe-subnav` / `.cooking-view-tabs` / 食材の表示切替）を隠しサイドバーに一本化。スマホは内部タブのまま。→ TKT-0157＋0158/0159/0160 に波及。
- スマホ下部タブは従来どおり3つ（食材管理/献立・レシピ/料理・記録）を温存。
- **献立スケジュール（TKT-0159）= 案A**: PCは「曜日=列/朝・昼・夜=行」の7日ウィークグリッドへ転置（既存の今日±3日ウィンドウ模型・±週ナビ・ドラッグ&ドロップ結線は不変、並べ方のみ変更）。スマホは現状の縦アジェンダを維持。サイドバーでは「献立・レシピ」グループ直下の葉として独立させる。
- **料理記録カレンダー（TKT-0160）**: 既に7列の月グリッドのため構造変更なし。大画面ではセル・写真サムネを拡大して見やすくするのみ（判断不要）。

### 理由
パレット・角丸・影・紫アクセントは既にモックと一致しており、作業の主眼は「見た目の作り替え」ではなく「PC用シェル（骨格）の追加」。スマホを温存して影響範囲を限定するため、デスクトップ表示はブレークポイントで分岐させる。
設定は使用頻度が低く主ナビの一等地を割く価値が薄いためサイドバー下部へ。ホームは在庫アラート・今日の献立・レシピ候補を一目で把握する価値があるためPC既定にするが、スマホは即タスクの軽快さを優先して据え置く。

### 方針の見直し（TKT-0138 を上書き）
TKT-0138 は「PCに広げず Canvas のモバイル型へ寄せる／独自のPC向けアレンジを入れない」方針だった。
本決定は **≥1024px のデスクトップ表示についてのみ**その方針を見直す（モバイル表示は TKT-0138 の意図どおり温存）。

### 却下した案
- 設定を主ナビ第4項目に昇格（案C） → スマホ下部タブが4つで窮屈、主ナビが薄まるため却下。
- ホームを独立させず食材管理上部に挨拶＋サマリーを差し込む（案C） → 最小実装だがモックの独立ダッシュボードと別物・拡張余地が小さい。軽量案として保留。
- ホームをPC/スマホ両方で初期表示（案D） → スマホで1タップ増、スマホ温存方針から外れるため却下。
- 献立スケジュールを縦アジェンダのまま幅だけ広げる（案B） → 低リスクだが大画面で横幅が余り「大画面化した感」が薄いため却下（保留）。
- 献立スケジュールに月カレンダー表示を新規追加（案C） → 俯瞰性は高いが7日ウィンドウ模型と別ロジックの新機能で、今回のレイアウト範囲を超えるため見送り（欲しければ別チケット化）。
- サイドバーIAをフラット昇格（案B: グループ見出しなしで葉を平坦に並べる） → 葉の所属（どのモードの機能か）が分かりにくく、スマホの3モード構造と対応が取りにくいため却下。
- サイドバーIAを3モードのまま（案C: サブタブはコンテンツ内に残す） → 最小実装だが大画面の縦空間を活かせず発見性も上がらないため却下（最軽量フォールバックとしてのみ保持）。
- アクション（＋追加/AI登録/レシピ生成等）も葉にする → ナビが操作で汚れ「目的地」と「操作」が混在するため却下。目的地のみ葉にする。

### 決定（TKT-0156: 消費量調整画面の調味料分類）
消費明細の食材/調味料分類は **レシピ材料の `item_type` を正本** とし、履歴編集画面
（`cooking-record-edit-modal.tsx`）もそれに合わせる。`ConsumptionEditDraft` に `item_type` を持たせ、
消費イベントは `材料名|単位` でレシピ材料に照合して引く（未一致は "食材" フォールバック）。
`cooking_consumption_events` への列追加（スキーマ変更）はしない。
AI生成（`recipe-generation.ts`）はプロンプトに調味料分類の明示指示＋スキーマ例の調味料行を追加する。

### 理由
履歴編集画面が「紐づく在庫アイテムの `category`」で分類していたため、在庫登録されない調味料は
`stockItem` が null → 既定 "食材" にフォールバックし、調味料タブが常に空だった。調理完了画面は既に
`ingredient.item_type` で正しく分類しており、2画面のロジック食い違いが混乱の元。レシピ材料を正本に
統一すれば、在庫の有無に依存せず・既存DBスキーマを変えずに両画面を一致させられる。

### 却下した案
- 「`cooking_consumption_events` に `item_type` 列を追加して消費確定時に保存」 → スキーマ変更（危険変更）で
  manual-smokes/review が重くなり、過去データの移行も必要。レシピ材料から引けば不要なため却下。

## 2026-06-02

### 決定（TKT-0151: Gemini AI利用の1日回数制限）
Gemini AI機能に日次上限（recipe 20 / scan 10 / 合計 30、JST暦日）をサーバー側で設ける。
判定は **原子的予約方式**: Gemini送信前に `consume_ai_usage()`（`pg_advisory_xact_lock` + 条件付きinsert）で
1回分を予約し、上限超過時はGeminiへ送らず429。Gemini通信失敗・写真DL失敗時は `refund_ai_usage()` で返金し
当日枠を消費しない。ok応答後のparse失敗は消費維持。`ai_usage_events` はRLS select本人のみ、
insert/update/delete は閉じ、書き込みは SECURITY DEFINER 関数経由（authenticatedのみ grant）。
refundは本人の直近5分の予約のみ削除可（履歴削除によるリセット不可）。上限値はSQL関数に単一定義。

### 理由
TKT-0150でAPIキーがユーザー持ちになり、上限の主目的は「運営者費用の抑制」→「乗っ取り・連打時の
本人課金の暴走防止の安全ネット」に変質した。連打耐性（原子的予約）と「一時的Geminiエラーで枠を
消費しない」（返金）を両立させるため、予約＋失敗時返金を採用。

### 却下した案
- 「成功時のみ記録（予約なし）」 → 大量同時リクエスト（乗っ取りバースト）に弱く、上限を一時的に超えうる。
- 「予約・失敗もカウント」 → 実装は最小だが、一時的502で自費キーの枠を消費しUX悪化。
- 上限値をlib/UIにも定義 → ドリフトの元。SQL単一ソースに統一。
- 日付境界UTC → DB実装は単純だがユーザー体験はJSTが自然。JSTを採用。

### 次に確認すること
- 公開前に `supabase db push` で適用し、実DBで上限到達・返金・別ユーザー分離・スマホ表示を手動確認。
- refundの5分制限が実Gemini最大応答時間と整合するか。
- `ai_usage_events` の行数増加に対する保全（定期削除）を別チケット化するか。
