# /plan-initiative — イニシアチブのラフ分割

新機能・大きめの変更を、**対話で要望を詰めた上で、複数チケットへラフに分割する**フェーズ。
詳細なチケットはここでは作らない。承認された分割案を `/breakdown` が詳細チケット化する。

> **必ず plan mode で使う。** 分割案は `ExitPlanMode` で提示し、ユーザー承認をハードな承認ゲートにする。
> 中間成果物（分割案）はファイル化せず会話コンテキストに保持する。承認後そのまま同一セッションで `/breakdown` へ渡す。

## 手順

1. **要望ヒアリング・対話**:
   - ユーザーの目的・背景・期待する完成像を聞く。曖昧な点は質問して詰める。
   - 既存状態を確認する: `project-os/backlog.md`（現在のフォーカス / 次にやる候補）、関連 `project-os/knowledge/decisions.md`。
   - 必要なら `web/` / `supabase/` の関連箇所を読み、実現可能性と影響範囲をざっと掴む。
2. **トレードオフと未確定点を正直に提示**し、対話で方針を収束させる。勝手に断定しない。
3. **ラフな分割案を作る**。各チケットは1行〜数行で、**詳細は書かない**:
   - 仮ID（`T1, T2, ...` でよい。実 TKT 番号は `/breakdown` が採番する）
   - title（一行）
   - goal（一行・何を防ぐ/実現するか）
   - 依存順（どれが先か。土台チケットを明示）
   - 危険度フラグ（🔴危険 / 🟢非危険）。危険変更の判定基準は下記。
   - 想定 changed_paths の当たり（`web/...` / `supabase/...` 程度の粒度でよい）
4. **分割案を `ExitPlanMode` で提示**する。plan の本文＝チケット分割案（上記の一覧）とし、
   「この構成で詳細チケット化に進んでよいか」を問う形にする。
5. 承認されたら実行モードへ移り、**続けて `/breakdown` を案内 or 実行**する
   （分割案は会話コンテキストに残っているので、そのまま詳細チケット化できる）。

## 危険度の判定基準（🔴危険変更）

`harness/change_evals.json` の `danger: true` に対応する。次に該当したら🔴を付ける:
- `supabase_schema_change`（DBスキーマ変更・migration）
- `auth_and_rls_policy`（認証・RLS ポリシー）
- `photo_upload_storage`（写真 Storage）
- `ai_server_route`（AI サーバー route）
- `csv_import_migration`（CSV 移行・データ削除/移行）

危険チケットは**独立したチケットに分離**し、非危険チケットと混ぜない（レビュー/manual-smokes が必要になるため）。

## 注意

- Canvas版 `app.html` は凍結。対象は Web版（`web/` + `supabase/`）のみ。
- この段階で詳細を作り込みすぎない。粒度確認・依存関係・危険度の切り分けに集中する。
- 分割案が小さすぎる（1チケットで足りる）と判断したら、無理に分割せず `/new-ticket` を案内する。
- 中間成果物はファイル化しない方針。**別セッションへ引き継ぐ必要が出たら**、ユーザー確認の上で分割案を
  `project-os/backlog.md` の「次にやる候補」に1行ずつ残すこと（INIT専用ファイルは作らない）。
